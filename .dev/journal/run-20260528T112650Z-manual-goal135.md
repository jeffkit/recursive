## Result

### `src/http.rs` (+126 lines)

- New `pub struct AuthConfig { keys: Arc<Vec<String>> }` with `Clone`,
  `Default`. Constructors: `new(Vec<String>)`, `default()`.
- `AuthConfig::is_valid(presented: &str) -> bool`: constant-time
  byte-by-byte comparison with OR-of-EQ accumulator. Loops over every
  configured key regardless of early match. Empty config returns
  `true` (auth-disabled semantics; middleware layer enforces the
  pass-through).
- `AuthConfig::is_enabled() -> bool`: non-empty key set.
- Private `auth_config_from_env()` reads
  `RECURSIVE_HTTP_AUTH_KEYS` (comma-separated, whitespace-trimmed,
  empty-filtered).
- `async fn auth_middleware`: pass-through when disabled; exempts
  `/health` and `/metrics` (k8s liveness + Prometheus); reads
  `X-API-Key` header; returns 401 with body `unauthorized` on
  missing/wrong key; otherwise calls `next.run(req)`.
- `pub fn build_router_with_auth(state, auth)`: new public
  constructor. Existing `build_router(state)` becomes a thin wrapper
  that calls `build_router_with_auth(state, auth_config_from_env())`.
- Layer order in `build_router_with_auth`: auth layer is added
  **after** rate-limit and metrics (axum applies in reverse), so it
  runs **first** — unauth requests are rejected before consuming
  rate-limit budget or polluting request counters.

### `tests/http.rs` (+129 lines)

Six new `#[tokio::test]`s inside `mod http_tests`, all using the new
`build_router_with_auth(state, AuthConfig::new(...))` injection point
to avoid `std::env::set_var` parallel-test races (per AGENTS.md
"Env-var tests must be ONE test, not many"):

- `auth_disabled_passes_through`: empty `AuthConfig` → /tools 200.
- `auth_enabled_rejects_missing_header`: 401 + body `unauthorized`.
- `auth_enabled_accepts_valid_key`: `X-API-Key: secret` → 200.
- `auth_enabled_rejects_wrong_key`: `X-API-Key: bogus` → 401.
- `auth_health_and_metrics_are_exempt`: both endpoints 200 even
  with auth enabled, no headers.
- `auth_config_is_valid_unit`: pure unit test for `is_valid` —
  empty config accepts everything; populated config rejects wrong
  case, off-by-one length (short and long), empty string, and
  unrelated key; accepts both configured keys.

   the new doc comment for `pub fn build_router` was appended directly
   to the existing routes-list doc without a `///` blank-line separator.
   Clippy interpreted "Build the public HTTP router..." as a malformed
   continuation of `- GET /openapi.json` list item. Resolved by merging
   the new auth-related sentence into the existing doc as a follow-up
   paragraph (with proper `///\n///` separator), reducing duplication
   instead of introducing it.

## Design decisions captured here

- **Layer order**: auth-first (in execution) is non-obvious in axum 0.8's
  reverse-application semantics. The spec called this out, and the
  implementation follows: auth `.layer()` is the **last** call in the
  chain so it executes **first**. Rate-limit and metrics layers
  thereafter only see authenticated requests.

- **`/health` and `/metrics` exemption**: hardcoded inside
  `auth_middleware`, not configurable. Rationale: in any real
  deployment, k8s liveness probes and Prometheus scrapers have
  separate, infrastructure-level access control. Forcing them to
  carry the API key creates friction for zero gain. This decision
  is documented in `AuthConfig`'s rustdoc.

- **Constant-time comparison hand-rolled**: did not pull in `subtle`
  crate. The OR-of-EQ accumulator with full loop iteration is
  ~10 lines and dependency-free. Tradeoff is documented; if a future
  goal needs more cryptographic operations (HMAC, signature
  verification), revisit.

- **`build_router_with_auth` as a public test seam**: introduced a
  second `pub fn` rather than `#[cfg(test)] pub` because the public
  HTTP API users may legitimately want to inject their own
  `AuthConfig` from a non-env source (config file, secrets manager).
  The seam has both a testing rationale and a production rationale,
  so it is unconditionally `pub`.

## Follow-ups recorded

- **g136 — Auth: JWT bearer tokens**: stack on top of `auth_middleware`.
  Likely structure: a `JwtConfig` alongside `AuthConfig`, a separate
  `Authorization: Bearer <token>` path inside the same middleware, and
  a unified `is_authorized(req: &Request)` helper. Will need
  `jsonwebtoken` crate.

- **Rate limiter test seam**: `rate_limiter_from_env()` has the same
  env-var coupling as `auth_config_from_env()` had before this goal.
  Worth introducing a similar `build_router_with_rate_limit(...)`
  injection point, but out of scope here.

- **Unused warnings on `build_router_with_auth` in non-test builds**:
  the new `pub fn` is only exercised by tests in this commit. Production
  callers (`src/main.rs::http_serve`) still call `build_router(state)`.
  This is fine — it is `pub`, so the binary won't dead-code-warn — but
  document it so a future cleanup pass doesn't yank it.
