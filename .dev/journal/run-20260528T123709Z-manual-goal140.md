g140 closes the wiring gap.

### `src/config_file.rs` (~25 lines)

- `FileConfig` gains `permissions: Option<PermissionsSection>`.
- New `PermissionsSection { allow, deny, interactive }` mirrors
  `PermissionsConfig` shape, lives in `config_file` to keep
  `permissions` module independent of TOML loading.

### `src/main.rs` (~50 lines)

- New `resolve_tool_permissions()` helper. Resolution order:
  1. `RECURSIVE_TOOL_PERMISSIONS_FILE=<path>` env — TOML file with
     top-level `allow`, `deny`, `interactive` keys (deserializes
     directly into `PermissionsConfig`).
  2. `~/.recursive/config.toml` `[permissions]` section.
  3. None — every tool allowed (back-compat).
- File read errors and TOML parse errors → `eprintln!` and treat
  as "no config" instead of failing the CLI. A malformed
  permissions file should not brick `recursive --help` or other
  unrelated subcommands.
- `build_tools` calls `registry.with_permissions(perms)` when the
  resolver returns `Some`.

### `tests/integration.rs` (~140 lines)

New `mod permissions` block:

- `permissions_deny_blocks_invoke` — explicit deny → `Error::PermissionDenied`.
- `permissions_allow_filter_blocks_unlisted` — non-empty allow
  list rejects unlisted tools.
- `permissions_glob_pattern_matches` — `deny=["run_*"]` blocks
  both `run_shell` and `run_background`; unrelated `read_file`
  passes through (verified by checking the error is NOT
  `PermissionDenied`).
- `permissions_no_config_allows_everything` — guards back-compat:
  a registry without `with_permissions` does NOT return
  `PermissionDenied` for normal calls.
- `permissions_section_parses_from_toml` — `FileConfig`
  deserializes a config.toml-style `[permissions]` block.

## Why this is "partial" 17.3, not full ✅

ROADMAP-v4 17.3 says "Tool permission system (role-based
allow/deny)". g133's `PermissionsConfig` is **global**: one rule
set applies to every caller. "Role-based" implies per-caller
context — different callers get different permissions, e.g. a JWT
with `role=admin` gets full access while `role=viewer` is
read-only.

Real role-based wiring requires:
1. Plumbing per-request context (the JWT claims, or some other
   identity token) into `ToolRegistry::invoke(...)`. The current
   signature is `invoke(&self, name, arguments)` — no caller
   context.
2. Some way to map roles → permission profiles (a config file
   structure, or a hook function).
3. The HTTP layer must extract role from the validated JWT (g136)
   and pass it down.

That coupling between auth and permissions is too big for a
single goal in the current shape. Defer to a future goal once a
real deployment surfaces a concrete need. Meanwhile:

- The global allow/deny shipped here covers the most common case
  ("never let the agent invoke shell" for an internal-tools
  deployment).
- Real per-deployment isolation is achievable today by running
  separate `recursive` instances with different
  `RECURSIVE_TOOL_PERMISSIONS_FILE` envs behind a router/proxy.

## Issues encountered & resolved

1. **`PermissionsConfig` already had `Deserialize`.** Initial
   sketch was going to add it; turns out g133 already did. Just
   needed to map `PermissionsSection` → `PermissionsConfig` in
   the resolver.

2. **`build_tools` couldn't unconditionally take a `PermissionsConfig`
   parameter** — it's called from many places (run_once, run_loop,
   repl, tests) and threading the option through all of them
   would have meant adding a 10th parameter to every call site.
   Instead, the resolver is a pure helper called from inside
   `build_tools` — sites that opt out of permissions just don't
   set the env or the config.toml section.

## Follow-ups recorded

- **Role-based per-caller permissions** (real 17.3 ✅): see "Why
  partial" above. Requires `ToolRegistry::invoke` to accept a
  caller-context parameter, or an alternative mechanism like an
  agent-level permission hook variant.
- **`is_interactive` is dead path.** g133 added it; today nothing
  consults it. The original idea was "ask user before running
  this tool" (REPL mode), but no caller does that. Either delete
  it or wire it up — separate goal.
- **`recursive permissions` CLI subcommand** to inspect / dry-run
  the active permissions config would be useful debugging UX. Out
  of scope.
