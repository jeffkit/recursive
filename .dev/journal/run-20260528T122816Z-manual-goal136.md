  falls back to `Authorization: Bearer <token>` for JWT

  401 only when both fail or the credentials are missing.
- `auth_config_from_env` reads two new env vars:
  `RECURSIVE_HTTP_AUTH_JWT_SECRET` (HMAC secret) and
  `RECURSIVE_HTTP_AUTH_JWT_AUDIENCE` (optional `aud` claim).

- `jwt_valid_token_accepted` — fresh token, no audience, accepted.
- `jwt_expired_token_rejected` — token with `exp` 5 min in the past

## Issues encountered & resolved

1. **`jsonwebtoken` default leeway masked the expired-token test.**
   First draft used `exp_offset_secs: -60`, which placed the token
   exactly at the boundary of `Validation::leeway` default of 60
   seconds. The decoder accepted it as not-yet-expired. Resolved
   by widening to -300 (5 minutes past) — well outside the leeway.
   Documented in the test comment.

2. **`jsonwebtoken` was not directly available to integration
   tests.** The `optional = true` + `dep:jsonwebtoken` setup
   exposes the crate to `src/` only when the http feature is on,
   but `tests/` is a separate compilation unit and cannot
   re-export through that gate. Fixed by adding
   `jsonwebtoken = "9"` under `[dev-dependencies]` so test files
   can `use jsonwebtoken::{encode, EncodingKey, Header}` directly.

## Design decisions captured here

- **HS256 only, no RSA/ECDSA in this revision.** Single env-var
  secret is operationally simpler than PEM key management. RSA
  becomes interesting when a deployment integrates with an
  external IdP via JWKS — a separate goal if/when needed.
- **`exp` mandatory.** RFC 7519 says `exp` is OPTIONAL. We
  override that via
  `Validation::set_required_spec_claims(&["exp"])`. Tokens without
  expiry are explicitly rejected — prevents accidentally minting
  immortal tokens that survive credential rotation.
- **Verify-only.** This server does not mint JWTs. Issuance is a
  separate concern (login flows, user management, refresh
  rotation). If/when the agent ever needs to call back into a
  service that requires the agent's own JWT, that's a different
  axis (outbound auth).
- **API key + JWT cohabit.** `AuthConfig::with_jwt` attaches a
  JWT verifier; existing API keys still work. This lets operators
  migrate gradually: legacy clients use X-API-Key, new clients
  use JWT, both go through the same middleware.
- **Header order tried: X-API-Key before Bearer.** API key
  comparison is constant-time byte comparison (cheap); JWT verify
  involves HMAC computation + JSON deserialization + claim
  validation. Cheap path first means denial-of-service via
  expensive crypto is harder.

## Follow-ups recorded

- **JWKS / RSA support**: when a deployment integrates with an
  external IdP (Auth0, Okta, etc.), it'll want to fetch a JSON Web
  Key Set and verify RS256/ES256 signatures. `jsonwebtoken`
  supports these — the constructor would just take a public key
  PEM and a different `Algorithm`. Out of scope.
- **Per-route auth granularity**: today auth is all-or-nothing
  (gate or pass through). A future goal might add a "this route
  requires `scope=admin`" rule, where the JWT's claims are
  inspected for fine-grained permissions. Tied to ROADMAP-v4 17.3
  ("Tool permission system, role-based"). Will revisit there.
- **Token minting / login endpoints**: not in this codebase yet.
  If we ever add `POST /login`, it'd live in a new
  `src/http/auth_endpoints.rs` module. Not on the immediate
  roadmap.
- **`exp` leeway tuning**: `jsonwebtoken`'s 60-second default is
  reasonable for clock skew across distributed systems. If a
  deployment finds it too lax, `Validation::leeway = 0` would
  enforce strict expiry. Currently un-exposed in `JwtConfig`;
  could add `JwtConfig::with_leeway(secs)` if requested.
