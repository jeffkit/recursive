## Result

ROADMAP-v4 17.1 was 🔴; g121 (Batch 37, 1d7a53d) had landed
`RateLimiter` + `rate_limit_middleware` with seven good unit tests
of the limiter logic itself, but no integration test ever sent a
real HTTP request through the axum middleware stack to verify the
end-to-end "exceed → 429" path.

g139 closes that with the same shape g135 used for auth: introduce
a public injection seam, then write integration tests against it.

### `src/http.rs` (~25 lines)

- `RateLimiter::new` made `pub` (was `fn`); doc comment expanded to
  mention the new injection use case.
- New `pub fn build_router_with_auth_and_rate_limit(state, auth,
  limiter)` — the lowest-level router constructor with both
  middleware states caller-supplied.
- `build_router_with_auth(state, auth)` becomes a thin wrapper
  around the new function with `rate_limiter_from_env()`.
  `build_router(state)` chain unchanged.

### `tests/http.rs` (~110 lines)

Four new `#[tokio::test]`s inside `mod http_tests`:

- `rate_limit_first_request_succeeds` — capacity=2, slow refill;
  first /health returns 200.
- `rate_limit_burst_allowed_then_429` — capacity=2: 1st and 2nd
  request to /tools succeed; 3rd returns 429 with body
  `"rate limit exceeded"`.
- `rate_limit_different_clients_have_independent_buckets` —
  capacity=1: alpha's 1st hit (200), beta's 1st hit (200, separate
  bucket), alpha's 2nd hit (429, alpha bucket exhausted).
- `rate_limit_does_not_block_below_threshold` — capacity=100, 5
  sequential hits all return 200.

All tests use `RateLimiter::new(capacity, 0.001)` with a refill
rate of 0.001 tokens/sec — slow enough that buckets cannot
replenish during the millisecond-scale test runtime. No
wall-clock sleeps; no env-var coupling.

## Pattern observation

Three goals in a row (g134 metrics, g135 auth, g139 rate-limit)
landed via the same pattern:
1. Production code from a prior goal was correct but lacked
   integration test coverage.
2. The integration tests required an injection seam that the
   prior goal hadn't created.
3. Solution: introduce a `build_router_with_<foo>(state, foo)`
   constructor parallel to the env-driven default; original
   `build_router` becomes a thin wrapper.

If a g140-and-later writes a NEW middleware, doing this seam from
day one is cheaper than retrofitting. Recommend: spec template for
HTTP middleware goals should require both an env-driven default
constructor AND an explicit-state constructor.
