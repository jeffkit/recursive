## Result

User reported: setting up provider via `recursive config set` worked
for the CLI but the TUI still answered `Error: no LLM provider
configured (set OPENAI_API_KEY or RECURSIVE_API_KEY)`. Root cause:
goal 143's `Backend::build_runtime` only read env vars and never
consulted `~/.recursive/config.toml`. Goal 149 closes that gap by
delegating to `recursive::config::Config::from_env()`, which already
implements the env > file > default priority chain CLI users expect.

### What landed

- `crates/recursive-tui/src/backend.rs::build_runtime` rewritten
  to call `recursive::config::Config::from_env()`, take
  `api_key`/`api_base`/`model`/`temperature`/`workspace` from the
  resolved `Config`, and surface a helpful offline reason that
  mentions `recursive config set` when no key can be found.
- `OpenAiProvider` now constructed with `with_temperature` to honour
  the resolved temperature (CLI does the same in `src/main.rs:405`).
- Module-doc updated to describe the new priority chain.
- The existing `offline_mode_returns_error_without_panic` test was
  renamed to `offline_mode_and_config_file_resolution` and extended
  to:
  - Pin HOME to a fresh tempdir via
    `recursive::test_util::PinnedHome` so we don't accidentally read
    the developer's real `~/.recursive/config.toml`
  - Part A: empty HOME + cleared env → assert offline error contains
    both `"no LLM provider configured"` and the new
    `"recursive config set"` hint
  - Part B: same HOME but with a valid `config.toml` → assert
    `build_runtime()` returns `RuntimeBuild::Ready`

### Decisions worth flagging

1. **Direct delegation to `Config::from_env`** instead of writing a
   parallel reader on top of `FileConfig` (what the goal §1 originally
   sketched). `Config` already implements: env > file > default with
   the right field-by-field merge, plus URL handling, plus retry
   policy, plus the memory-layer system prompt. Re-implementing any
   of that in the TUI would be brittle. The trade-off: TUI now
   transitively wires through some CLI-specific behaviour like
   memory-layer prompt assembly, which is fine — those layers are
   no-ops when the workspace has no `.recursive/memory/` files.
2. **No new dep**, `recursive::config::Config` is already public.
3. **`provider_type != "openai"` (e.g. anthropic)** is not yet
   handled in the TUI — we always construct `OpenAiProvider`. CLI
   has an Anthropic branch (`src/main.rs:393`) that we could mirror
   later, but Goal 149's scope is "make the user's existing
   DeepSeek/OpenAI config work in the TUI", not extend the provider
   matrix. If a future user reports the same gap for Anthropic,
   that's a separate goal.
4. **Test consolidation**: Part A + Part B sit in the same test
   function so they share the `PinnedHome` lock. Splitting them
   into two `#[test]` fns would race on HOME (cf. lesson 17).
5. **Environment cleanup is unconditional** — Part A removes
   `RECURSIVE_API_KEY` / `OPENAI_API_KEY` and Part B doesn't put
   them back until the very end, so a developer running tests in a
   shell with those vars set sees their values restored after the
   test. The `_pin` guard restores HOME on drop the same way.
