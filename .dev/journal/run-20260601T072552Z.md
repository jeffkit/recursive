for the full spec.

## What was done

### 1. `Tool` trait — `should_defer()` and `search_hint()`

`src/tools/mod.rs:66` — added two new methods to the trait, both with
`false` / `None` defaults. No existing `Tool` impl needed to change;
the deferred loading is opt-in.

The goal spec put `search_hint` on `ToolSpec` (the wire shape), but
I moved it to the `Tool` trait because:

- `search_hint` is *local* metadata for the search engine, never
  sent to the model. Putting it on `ToolSpec` pollutes the wire
  shape (29 existing `ToolSpec { ... }` literals all break).
- The `(spec, hint)` pair is what the search engine actually needs;
  the registry collects both via `(tool.spec(), tool.search_hint())`.
- This is consistent with `should_defer()` — a property of the tool,
  not of its wire description.

Net effect: zero changes to existing tool impls, zero changes to
the `ToolSpec` wire shape.

### 2. `ToolRegistry::partition_by_eagerness`

`src/tools/mod.rs:258` — new method that splits a registry into
`(eager, deferred)` based on:

- `always_eager` whitelist (caller-supplied; not hard-coded)
- `should_defer() == true`

The whitelist is passed in by the agent (`ALWAYS_EAGER` const in
`agent.rs:557`), so the per-provider policy is decided at the agent
layer, not in the registry.

Added two unit tests in `src/tools/mod.rs`:

- `partition_by_eagerness_splits_on_should_defer` — verifies a tool
  with `should_defer: true` (with and without a hint) ends up in
  `deferred`, and that the `search_hint` flows through to the
  deferred list.
- `partition_by_eagerness_whitelist_overrides_should_defer` —
  verifies a tool in the `always_eager` list lands in `eager`
  even if it returns `true` from `should_defer()`.

### 3. `LlmProvider::complete_with_search`

`src/llm/mod.rs:313` — new trait method with a default impl that
concatenates `(eager + deferred)` and calls `complete()`. The
default impl is the entire story for `OpenAiProvider` and
`MockProvider`: byte-for-byte identical behavior, no code change.

`AnthropicProvider` overrides it (see #5 below).

### 4. `src/llm/search.rs` — new module

Ported the keyword search algorithm from
`fake-cc/src/tools/ToolSearchTool/ToolSearchTool.ts:186-302`:

- `ToolSearchEngine` trait with `resolve(&str, &[SpecWithHint]) -> Vec<String>`.
- `KeywordSearchEngine` is the default impl. Pure function — no I/O,
  no async — so it's trivial to test.
- Two-phase algorithm: exact name match (case-insensitive) → scored
  keyword search with `search_hint` (+4), name part (+10 MCP / +10
  regular), description word-boundary (+2).
- The `+` prefix marks a term as required.
- `mcp__server` prefix matching for MCP tools.
- `MAX_RESULTS_CAP = 20`, default = 5.

Added 16 unit tests covering all paths: exact match, case
insensitivity, MCP prefix, `searchHint` weighting, required terms,
empty/whitespace queries, max_results cap, CamelCase splitting,
word-boundary false positive prevention, no-match.

### 5. `AnthropicProvider::complete_with_search` override

`src/llm/anthropic.rs` — added:

- `search_engine: Arc<dyn ToolSearchEngine>` field + `with_search_engine`
  builder.
- `ToolSearchTool` spec (hard-coded; same wire shape as fake-cc's).
- `run_search_aware_loop` — the recursive loop. On each round:
  1. Build the request body with eager (no `defer_loading`) +
     `ToolSearchTool` + deferred (each marked `defer_loading: true`).
  2. Send and parse.
  3. If the model called `ToolSearchTool`, resolve the query
     against the deferred candidates, append an assistant message
     + a marker user message, and recurse with
     `pending_tool_result = Some((id, names))`.
  4. The body builder rewrites the marker message into a proper
     `tool_result` content block whose `content` is an array of
     `tool_reference` items.
- `build_request_with_partition` — request body builder. Two new
  helpers (`tool_reference_array`) emit the wire shape.
- `MAX_SEARCH_ROUNDS = 3` cap on the recursion (prevents the model
  from looping on search calls).

`post_with_retry` was extracted from `complete()` to share between
the non-streaming search path and (in the future) the streaming
path.

#### Key implementation note

The `tool_reference` round-trip is structurally subtle. The
Anthropic API requires `tool_result.content` to be either a string
(text) or an array of content blocks. To get `tool_reference` items
into the result, the array form is required. So the body builder
overwrites the entire last message (not just its `content` field) —
the marker user message is replaced with a hand-rolled
`{"role":"user","content":[{"type":"tool_result","tool_use_id":...,
"content":[{"type":"tool_reference","tool_name":...}]}]}` JSON
object. This was the cleanest way to do it without extending
`Message` or `serialize_message`.

#### End-to-end mock test

`src/llm/anthropic.rs:1789` — `complete_with_search_round_trips_tool_reference`.
Spawns a mock TCP server that:

1. On the first request, returns a `ToolSearchTool` tool_use with
   query "jupyter" and max_results=3.
2. On the second request, returns a real `notebook_edit` tool_use.

Asserts:

- Two requests were sent (the search round-trip happened).
- The first request's `tools` array has all three names
  (`ToolSearchTool`, `read_file`, `notebook_edit`) with
  `notebook_edit` marked `defer_loading: true`.
- The second request's last message is a `tool_result` block whose
  `content` is a non-empty array of `tool_reference` items.
- The final `Completion` is the `notebook_edit` call.

Also added two unit tests on the body builder:

- `build_request_with_partition_marks_deferred_tools` — eager
  tools have no `defer_loading` key, deferred tools have
  `defer_loading: true`.
- `build_request_with_partition_rewrites_pending_tool_result` —
  the last message is replaced with a `tool_result` block whose
  `content` is an array of `tool_reference` items.
