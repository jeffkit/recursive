remains non-functional in this orchestrator's environment (root cause
still uninvestigated; tracked across prior journals).

64dfec1, "merge: goal-120 graceful-shutdown … runtime integration
deferred"). That landing wired SIGINT/SIGTERM handling and added
`FinishReason::Cancelled`, but never connected the resulting
`CancellationToken` to the agent loop. The token was inspected only

behavior change. `Agent::shutdown_token` (added by g120) and

grep of `src/` for `FinishReason::Cancelled` constructors returned
zero hits.

g137 closes this gap.

## Result

### `src/agent.rs` (~45 lines)

- `RunCore` struct (line 273): new field
  `pub(crate) shutdown_token: Option<CancellationToken>`.
- `RunCore::run_inner` step loop (line 558): new termination block at
  the top of `for step in 1..=self.max_steps`. Emits
  `StepEvent::Finished { reason: FinishReason::Cancelled, steps:
  step.saturating_sub(1) }`, logs at info level mirroring the existing
  finish-log pattern, and returns `RunInnerOutcome` with all in-flight
  state preserved (messages, latency, plan_buffer, plan_confirmed).
- `Agent::run` (line 1045): `RunCore` constructor now forwards
  `self.shutdown_token.clone()`. The `Agent` struct's existing
  `shutdown_token` field and `set_shutdown_token` setter (deprecated
  path, src/agent.rs:253/870) become live.

The check uses `step.saturating_sub(1)` for the step count: if
cancellation is observed at the top of step N, the agent has
completed N-1 full steps. The in-flight LLM call from step N-1
finished and was processed before the next iteration began; we never
started step N's LLM call.

## Issues encountered & resolved

1. **Test B initial failure: anti-stuck masked Cancelled.** First
   draft scripted the same `__noop__` tool with empty args 4 times.
   `agent.rs::run_inner` line 807 detected three identical-key error
   results in a row and finished with
   `FinishReason::Stuck { repeated_call: "__noop__", repeats: 3 }`,
   beating cancellation to the punch. Fixed by giving each scripted
   tool call a distinct `tag` arg (anti-stuck keys on `(name, args)`
   tuple, not just name).

2. **Test B second failure: Notify timing race.** With a single-thread
   tokio runtime (default for `#[tokio::test]`), the canceller's
   `notified()` future was registered AFTER MockProvider's
   `notify_one()` had already fired, dropping the permit. Result:
   token never cancelled, agent ran the script to completion. Fixed
   in two ways:
   - Switched to `multi_thread` flavor so the canceller and the
     agent run on independent worker threads.
   - Pre-registered the `notified()` future before spawning the
     agent: the canceller signals "I am armed" via a second Notify,
     the test awaits that signal, only then does `runtime.run()`
     start. This removes any timing assumption.

3. **Test D / E compile errors after first integration build.**
   Three issues in initial test code:
   - `AgentKernel.shutdown_token` is `pub(crate)` so tests outside
     the crate cannot read it. Fixed by adding a public
     `shutdown_token() -> Option<&CancellationToken>` getter on
     `AgentKernel` (test seam, also useful for production callers
     that want to check token state).
   - `AgentBuilder::hooks(...)` does not exist — the legacy builder
     uses singular `.hook(Arc<dyn Hook>)`. Updated Test E to call
     `.hook(counter.clone())` directly, dropping the unused
     `HookRegistry::new()` ceremony.
   - The current `Hook` trait's `on_event` is non-async and takes
     `HookEvent` by value, not `&HookEvent<'_>` — my initial
     async_trait sketch was wrong. Updated to the actual sync
     signature.

## Design decisions captured here

- **Termination check at loop head, not inside step body.** The
  invariant #1 spirit (no new capabilities inside the loop) is
  preserved: cancellation is a termination condition, structurally
  identical to the existing `BudgetExceeded` and `TranscriptLimit`
  blocks. None of them call user code or invoke tools — they observe
  state, log, and return. Adding a fourth member to that family is
  an extension along the same axis, not a new dimension.

- **Token lives on AgentKernel, not AgentRuntime.** The token is
  process-level singleton-shaped (Clone via Arc), not per-turn
  state. Putting it on `AgentKernel` means `with_tools`-cloned
  sub-agents inherit it for free, and the `RunCore` construction
  point already has access. `AgentRuntimeBuilder::shutdown_token`
  is a delegation convenience.

- **HTTP server unchanged.** `axum::serve(...).
  with_graceful_shutdown(...)` (src/main.rs:386) operates at the
  listener level and was already correct in g120. Per-run agent
  cancellation inside HTTP session handlers is not wired by this
  goal — they pass `None` to `build_runtime`. Rationale: HTTP
  clients that want to abandon a long-running agent run already
  have the protocol-level mechanism (drop the connection, or
  DELETE the session). Wiring per-request cancellation through the
  HTTP layer is a distinct axis and a separate goal if the need
  arises.

- **REPL pass-through.** `repl()` passes `None` for the shutdown
  token. REPL's `Ctrl-C` semantics belong to the readline frontend
  (cancel current input line), not to the agent loop. If a future
  goal wants graceful agent cancellation inside REPL, a different
  shape is needed (e.g. a per-turn token reset).

## Follow-ups recorded

- **Kernel path SessionEnd hook dispatch**. `Agent::run` dispatches
  `HookEvent::SessionEnd` (gated to NoMoreToolCalls / Stuck /
  BudgetExceeded), but `AgentKernel::run` (the modern path used by
  `AgentRuntime`) does not dispatch it at all. This is a pre-existing
  gap, surfaced but not closed by g137. Test E only validates the
  legacy path. A follow-up goal could move dispatch into the kernel
  to unify behavior.

- **HTTP per-session cancellation**. As above — out of scope here,
  but a clean future goal: `DELETE /sessions/{id}` could push a
  cancellation through the session's running `AgentRuntime`. Today
  it removes the SessionState entry but doesn't interrupt an
  in-flight agent loop on that session.

- **`tests/integration.rs` heavily uses deprecated `Agent`**. The
  file's top-of-file note says "tests are not fully migrated".
  g137 added a `mod shutdown` block in the modern `AgentRuntime`
  style; new shutdown-adjacent tests should follow that pattern.
