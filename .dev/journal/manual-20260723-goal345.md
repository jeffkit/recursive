**Notes**:
- The compaction safety check (`MIN_CONVERSATIONAL_IN_OLDER = 2`) rejects compactions where the older slice has fewer than 2 User/Assistant messages. This directly prevents the g331 failure mode (system-prompt-only summary → "No conversation found to summarize").
- The `safe_split_point` now has a forward pass that advances past any System messages at the split boundary, ensuring the system prompt always stays in the kept segment.
- The wall-clock exit stores `wall_timeout_secs` and `wall_start` on `RunCore`. The deadline check runs at each step boundary (alongside shutdown/transcript-budget checks) and produces `FinishReason::WallClockExceeded` (data, not an error — per invariant #7).
- CLI flag `--wall-timeout` and env `RECURSIVE_WALL_TIMEOUT_SECS` are parsed in `Config::from_env()` but not yet wired to the CLI argument parser. The default is 0 (unset).
- The flow signal handler fix prevents the g331 data-loss pattern: when SIGINT/SIGTERM arrives before a terminal verdict, `preserveScene` is called instead of `cleanupWt`. If `preserveScene` fails (signal context is hostile), the worktree is left in place rather than force-removed.
- 3 existing tests needed adjustment for the new compaction safety: `apply_to_transcript_minimum_length_boundary`, `apply_to_transcript_boundary_plus_not_times`, `hooks_and_compaction`, and `compaction_summary_appears_in_jsonl`. All were updated to account for the `MIN_CONVERSATIONAL_IN_OLDER` check.
- **Follow-on fix (post-preserve, by the human-in-the-loop)**: the original `apply_to_transcript` degenerate-slice rejection made `maybe_compact` (run_core.rs) and `maybe_compact_cross_turn` (runtime.rs) dispatch `PreCompact` *before* calling `apply_to_transcript`, so the new `Ok(None)` path fired `PreCompact` with no matching `PostCompact` — breaking `tests/integration.rs::hooks_and_compaction` (`PreCompact=2, PostCompact=0`). Fixed by extracting a pure `Compactor::would_compact` predicate (single source of truth shared with `apply_to_transcript` via `split_for_compaction`) and gating `PreCompact` on it in both callers, so the hooks stay balanced (both fire, or neither does). This was the gate-test blocker that kept the run in `failed-preserved`; after this fix `cargo test --workspace` is green.
- **Flaky-test hardening (post-preserve, by the human-in-the-loop)**: the first `--land-preserve` run failed `land.gate.test` on `logging::tests::guard_restores_quiet_state_on_drop` (`assert!(!is_tui_quiet())` at `src/logging.rs`). Root cause: `TUI_QUIET` is a process-global `AtomicBool` and all 7 tests in `logging::tests` mutate it; run in parallel (cargo test default) they race — one test's `set_tui_quiet(true)` flips the flag mid-`assert!(!is_tui_quiet())` in another. Pre-existing flakiness, unrelated to Goal 345, but it blocked landing. Fixed by adding a module-local `static QUIET_TEST_LOCK: Mutex<()>` and acquiring it at the top of every test in `logging::tests`, serializing the whole module. Verified by running `cargo test --lib logging::` 5× (all green) and `cargo test --workspace` (green).
- **TUI presence-gate opt-out (post-preserve, by the human-in-the-loop)**: the second `--land-preserve` run passed test/clippy/fmt/e2e but failed `land.gate.tui-presence` because `crates/recursive-tui/src/runtime_builder.rs` changed with no test-bearing change. The change is a single line — `wall_timeout_secs: 0,` added to a `Config` struct literal — a pure mechanical field init to keep the construction compiling after `Config` gained the field in `src/config.rs`. Value `0` means wall-timeout disabled (unset), and the field is not consumed by the TUI runtime at all (the wall-clock deadline lives on `RunCore`, not the TUI). Zero behaviour change → no TUI test is meaningful (a tautological test would be rejected by `tui-mutants`). Per `.dev/skills/tui-acceptance.md` / `CLAUDE.md`, this is the sanctioned `RECURSIVE_TUI_TEST_PRESENCE=0` opt-out case for a pure refactor; the `--land-preserve` re-run is launched with that env var set.
- **CLI presence-gate opt-out (post-preserve, by the human-in-the-loop)**: the third `--land-preserve` run passed every gate through `agent-mutants` but failed `land.gate.cli-presence` because `crates/recursive-cli/src/main.rs` changed with no test-bearing change. Same root cause as the TUI one: the only change is `wall_timeout_secs: 0,` added to a `Config` literal inside `mod tests` (a test-helper construction), purely to keep it compiling after `Config` gained the field. Zero behaviour change (value `0` = disabled, field not exercised by any CLI path). Sanctioned `RECURSIVE_CLI_TEST_PRESENCE=0` opt-out; the `--land-preserve` re-run is launched with both `RECURSIVE_TUI_TEST_PRESENCE=0` and `RECURSIVE_CLI_TEST_PRESENCE=0` set.
