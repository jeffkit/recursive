# Goal 273 — Reason about reasoning_tokens in cost

**Roadmap**: Phase 17 (Production Hardening) — P1 from
`docs/review/architecture-review-2026-06-10.md` (NEW-COST-1)

**Design principle check**:
- Implemented as: add `reasoning_tokens: u32` to `TokenUsage`,
  accumulate in `SessionCost`, plumb from `UsageMeta`, surface
  in `cost.json`
- ❌ Does NOT branch inside `agent.rs::Agent::run`'s main loop
- ❌ Does NOT add a new feature flag

## Why

`TokenUsage` (src/llm/mod.rs:82) tracks `prompt_tokens`,
`completion_tokens`, `cache_hit_tokens`, `cache_miss_tokens` —
but **not** `reasoning_tokens`. DeepSeek R1 and OpenAI o1
emit reasoning tokens as a separate stream, often 10-50× the
completion count for long R1 sessions. `SessionCost::accumulate`
(src/session.rs:241) only sums input + output + cache, so
reasoning tokens are silently dropped from the cost total.

The provider code (src/llm/openai.rs:408) already reads
`completion.reasoning_content` and stores it in `Message`, but
no path counts the tokens. End users running R1/o1 see
`cost.json` totals that are 10-50× lower than reality.

## Scope (do exactly this, no more)

### 1. Add `reasoning_tokens` to `TokenUsage`

In `src/llm/mod.rs`, modify the `TokenUsage` struct (line 82)
and its `accumulate` method:

```rust
pub struct TokenUsage {
    pub prompt_tokens: u32,
    pub completion_tokens: u32,
    pub total_tokens: u32,
    pub cache_hit_tokens: u32,
    pub cache_miss_tokens: u32,
    /// Reasoning/thinking tokens emitted by models that support
    /// extended thinking (DeepSeek R1, OpenAI o1, Anthropic
    /// extended thinking). Adds to the cost total because the
    /// model spent compute producing them.
    pub reasoning_tokens: u32,
}

impl TokenUsage {
    pub fn accumulate(self, other: TokenUsage) -> TokenUsage {
        TokenUsage {
            prompt_tokens: self.prompt_tokens + other.prompt_tokens,
            completion_tokens: self.completion_tokens + other.completion_tokens,
            total_tokens: self.total_tokens + other.total_tokens,
            cache_hit_tokens: self.cache_hit_tokens + other.cache_hit_tokens,
            cache_miss_tokens: self.cache_miss_tokens + other.cache_miss_tokens,
            reasoning_tokens: self.reasoning_tokens + other.reasoning_tokens,
        }
    }
}
```

### 2. Wire it through `UsageMeta`

`UsageMeta` (src/session.rs:241) maps `TokenUsage` into
the session-cost shape. Add the field and the
`from_token_usage` mapping. Default 0 for sessions without
reasoning (pre-Goal-273 sessions will see 0).

### 3. Sum it in `SessionCost::accumulate`

In src/session.rs::SessionCost::accumulate (around line 290),
add `reasoning_tokens` to the running total:

```rust
SessionCost {
    /* ... existing fields ... */
    total_reasoning_tokens: u64,
}
```

The `accumulate` method should add the new field too. If
`CostData` serializes `reasoning_tokens` to `cost.json`, add
the field there as well.

### 4. Pricing math

The simplest correct choice: treat `reasoning_tokens` as
output tokens for pricing purposes (they were generated by
the model and cost compute). The `cost_usd()` function in
`cost.rs` should add a `* output_per_million / 1M` term for
reasoning tokens:

```rust
fn cost_usd(&self, usage: TokenUsage) -> f64 {
    let p = self.pricing;
    let input = usage.prompt_tokens as f64 * p.input_per_million / 1e6;
    let output = (usage.completion_tokens + usage.reasoning_tokens) as f64
        * p.output_per_million / 1e6;
    /* ... cache math ... */
    input + output + cache_discount
}
```

If the existing `cost_usd` is a method on a different type,
match its structure. The key point: `reasoning_tokens` adds
to the output bill.

### 5. Provider integration

In src/llm/openai.rs (line ~408 where `reasoning_content`
is read from the completion), if the OpenAI/Anthropic
provider already reports reasoning token counts via a
separate API field (e.g. `usage.completion_tokens_details.reasoning_tokens`),
populate `TokenUsage::reasoning_tokens` from there.

If the provider does NOT report a separate count (the
common case for non-reasoning models), default to 0 — the
field exists but is unused, which is fine.

**Do NOT** try to estimate reasoning token count by
`reasoning_content.len() / 4` or similar heuristic. That
introduces a maintenance burden and a likely off-by-some-
factor bug. If the provider doesn't report it, leave it 0.

## Notes for the agent

- The new field is `reasoning_tokens: u32` (matching
  `prompt_tokens` / `completion_tokens`). NOT `u64` — these
  are per-call counts, u32 is plenty (4B tokens is 1T context
  windows).
- `SessionCost` totals are `u64` (cumulative). Match that
  pattern.
- For the OpenAI provider, the relevant API field is
  `usage.completion_tokens_details.reasoning_tokens` (o1
  family). It may or may not be present; default 0 is fine.
- Do NOT add `reasoning_tokens` to the `Total tokens` field
  on `TokenUsage` — `total_tokens` is a provider-reported
  field, not a derived sum. Reasoning tokens are a separate
  bucket that adds to cost, not to the conversation-context
  count.
- Estimated diff: 3 files (mod.rs, session.rs, cost.rs) +
  1 file (openai.rs, only if the API field is trivial to
  extract). ~50-100 lines net.
- **Test discipline reminder (from g268 post-mortem)**: use
  deterministic serde / direct field tests, not runtime.
  `TokenUsage::default()` returns 0 for all fields; build
  test fixtures with `..default()` syntax.

**Disjoint file guarantee**: This goal touches

src/llm/openai.rs. Goal 272 touches src/http/auth.rs and
