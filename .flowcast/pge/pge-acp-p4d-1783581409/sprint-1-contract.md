# Contract: Sprint 1 — Foundation: stdio handshake + text-only agent loop (rev 2)

Establish the ACP stdio JSON-RPC transport and text-only agent loop. The subcommand `recursive acp` reads line-delimited JSON-RPC 2.0 requests from stdin, writes responses and notifications to stdout, and routes all logging to stderr. Supports initialize handshake with full capabilities declaration (8 forward-capability keys; terminalCapabilities explicitly false) followed by an `initialized` notification, session/new with sandbox-rooted working directory and sequential session IDs (`acp-sess-N`), and session/prompt accepting ContentBlock[] input and streaming `session/update` notifications (`agent_message_chunk` + terminal `end_turn` with stopReason). Internal architecture: ACP adapter is a thin EventSink→notification translator over AgentRuntime::run(), never touching run_inner.

## Criteria
- [AC-S1-01] `recursive acp` subcommand starts, reads a JSON-RPC 2.0 initialize request from stdin (one line), and writes exactly one JSON-RPC response line to stdout containing `result.protocolVersion` (integer 1), `result.agentInfo.name` (string "recursive"), and a valid `result.agentCapabilities` object with 8 keys. A second line with an `initialized` notification follows.
  - how: Run `echo '{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":1,"clientCapabilities":{},"clientInfo":{"name":"test","version":"0.1"}}}' | recursive acp`. Assert stdout has at least 2 lines. Line 1: valid JSON with `jsonrpc:"2.0"`, `id:1`, `result.protocolVersion == 1`, `result.agentInfo.name == "recursive"`, and `result.agentCapabilities` with ≥5 keys including all 8 required capability keys. Line 2: valid JSON-RPC notification with `method:"initialized"`. Assert exit code 0.
- [AC-S1-02] All logging, tracing, and diagnostic output goes exclusively to stderr. No protocol-invalid bytes (non-JSON, unstructured text) appear on stdout during any request/response cycle.
  - how: Run `echo '...initialize request...' | recursive acp 2>stderr.log`. Capture stdout to a file. Assert stdout contains only valid JSON-RPC 2.0 objects, one per line, and no unstructured text lines. Assert stderr.log is non-empty (contains log lines). No cross-contamination.
- [AC-S1-03] `session/new` accepts a `cwd` (absolute path) parameter and returns a unique-per-invocation session ID. The session IDs are sequential (`acp-sess-1`, `acp-sess-2`, …) and never collide, even when two session/new calls use identical cwd.
  - how: Call `session/new` twice with identical `{"cwd":"/tmp/test-sandbox"}`. Assert both responses return different `sessionId` values (both matching the pattern `acp-sess-\d+`). Assert the second ID is lexicographically greater than the first. Call again with a different `cwd` and assert yet another unique ID.
- [AC-S1-04] `session/new` maps the provided `cwd` through canonicalization to establish the sandbox root. The session is anchored to the resolved path and all subsequent filesystem operations are bounded by it.
  - how: Call `session/new` with `cwd` pointing to a directory inside the workspace. Then call `session/prompt` with a prompt like "list files in /etc" — the agent's Bash tool must be sandboxed (cannot read /etc/passwd). Call `session/new` with `cwd` pointing to a non-existent path and assert the request returns a JSON-RPC error (code -32602). Call `session/new` with `cwd` pointing to a file (not a directory) and assert a JSON-RPC error (code -32602).
- [AC-S1-05] `session/prompt` accepts a `sessionId` and a `prompt` (ContentBlock[] per ACP spec — an array of `{"type":"text","text":"..."}` blocks) and, after validation, begins streaming `session/update` notifications to stdout — one JSON-RPC notification per line — as the agent generates its response.
  - how: After creating a session, send `session/prompt` with a simple prompt like `{"prompt":[{"type":"text","text":"Hello, what is 2+2?"}]}`. Assert stdout contains one or more lines matching `{"jsonrpc":"2.0","method":"session/update","params":{...}}`. Each notification must be on its own line. The `params` must include `sessionId` matching the created session and `messageId` (a 12-character lowercase hex string from SHA-256).
- [AC-S1-06] The final `end_turn` notification for a `session/prompt` response carries a `stopReason` field whose value is one of `end_turn`, `max_turns`, `cancelled`, or `error`. The `session/prompt` JSON-RPC response (the `result` line) contains a matching `stopReason`. No further notifications for that turn are emitted after the `end_turn`.
  - how: Send `session/prompt` with a prompt that the agent can complete within budget. Assert the terminal notification has `update.sessionUpdate:"end_turn"` and `update.stopReason:"end_turn"`. Send another prompt with `maxSteps:1` and assert the terminal notification has `stopReason:"max_turns"`. In both cases, assert the `session/prompt` JSON-RPC response (the `result` line) arrives only after the `end_turn` notification has been emitted, and its `stopReason` matches the notification's value.
- [AC-S1-07] `session/prompt` is text-only in this sprint. No `tool_call` or `tool_call_update` notifications (with `status: pending/in_progress/completed`) are emitted on stdout during a `session/prompt` cycle.
  - how: Send `session/prompt` with a prompt likely to trigger tool use (e.g., "list files in the current directory"). Assert that stdout contains no JSON-RPC notification lines with `update.sessionUpdate` equal to `tool_call` or `tool_call_update`. The agent may internally use tools, but the ACP adapter must not emit tool-call lifecycle notifications in Sprint 1.
- [AC-S1-08] The protocol transport is line-delimited: each JSON-RPC message (request, response, or notification) is exactly one line. No pretty-printing, no multi-line JSON, no trailing whitespace.
  - how: Run the full initialize→session/new→session/prompt cycle. For every line on stdout, assert: (a) the line is valid JSON, (b) the line contains no embedded newline characters, (c) `jsonrpc` field equals `"2.0"`. Use `wc -l` on stdout and count JSON objects via `jq` to verify 1:1 correspondence.
- [AC-S1-09] The initialize response declares `agentCapabilities` that accurately reflect what the server supports in this sprint. All 8 canonical ACP keys are present, with only `terminalCapabilities` explicitly set to `false` (all others are truthy — non-null, non-false — as forward declarations).
  - how: Parse the initialize response's `result.agentCapabilities`. Assert all 8 keys exist: `promptCapabilities`, `toolCallNotifications`, `loadSession`, `resume`, `fs.readTextFile`, `fs.writeTextFile`, `mcpCapabilities`, `terminalCapabilities`. Assert `terminalCapabilities` is exactly `false`. Assert all other 7 keys are truthy (non-null and not `false`).
- [AC-S1-10] `session/prompt` returns a JSON-RPC error when the referenced `sessionId` does not exist or was never created, with an error code in the protocol error range (-32000 to -32099).
  - how: Send `session/prompt` with a fabricated `sessionId` (e.g., `acp-sess-999` that was never created). Assert the response is a JSON-RPC error (contains `error` field, no `result`), with `error.code` in range -32099..=-32000, and `error.message` indicating the session was not found.
- [AC-S1-11] The ACP adapter code does not import, reference, or branch on any symbol from `src/run_core.rs::RunCore` or `run_inner`. It interacts with the agent exclusively through `AgentRuntime::run()` and the `EventSink` trait.
  - how: Run `grep -r 'run_core\|run_inner\|RunCore' src/acp/`. Assert zero matches. Review the ACP adapter source: every code path that triggers agent execution must go through `AgentRuntime::run()` or a public wrapper thereof, never through the low-level run loop.
- [AC-S1-12] The `messageId` assigned to each `agent_message_chunk` and `end_turn` notification is stable and deterministic: it is the first 12 hex characters of the SHA-256 hash of the accumulated response text. Two sessions sending the same prompt produce the same `messageId` (though different `sessionId`).
  - how: Create two separate sessions with different `cwd` values. Send the identical prompt `[{"type":"text","text":"Hello"}]` to both. Assert that the `params.update.messageId` field in the first agent_message_chunk notification of each session is identical (a 12-char lowercase hex string), confirming content-hash determinism independent of session identity.
- [AC-S1-13] The server processes requests sequentially: a second JSON-RPC request sent on stdin while a `session/prompt` is still streaming is not read or processed until after the first request's response has been written to stdout.
  - how: Pipe two requests in quick succession: initialize, then immediately session/new. Assert that the response order matches request order — the initialize result arrives before the session/new result. Assert no interleaving of response lines.
- [AC-S1-14] Malformed JSON-RPC requests (invalid JSON, missing `jsonrpc` field, missing `method` field on non-notification requests) produce a JSON-RPC error response with code -32600 (Invalid Request) or -32700 (Parse error) and do not crash the server process.
  - how: Pipe `not json` into `recursive acp`. Assert stdout contains a JSON-RPC error with code -32700. Pipe `{"id":1}` (missing jsonrpc version). Assert error with code -32600. In both cases, the server remains running and can process a subsequent valid `initialize` request.
- [AC-S1-15] After a successful `initialize` handshake, the server emits an `initialized` notification as the second stdout line. This notification has `method:"initialized"` and `params:{}`. It appears after the initialize response and before any subsequent responses or notifications.
  - how: Run the initialize handshake. Assert the second stdout line is a valid JSON-RPC notification with `jsonrpc:"2.0"`, `method:"initialized"`, and no `id` field. Assert it appears immediately after the initialize response line (line 2).
- [AC-S1-16] `session/prompt` returns a JSON-RPC error (code -32602) when `prompt` is missing, is not a valid ContentBlock[] array, or is an empty array with no text blocks.
  - how: Send `session/prompt` without a `prompt` field. Assert error code -32602. Send `session/prompt` with `prompt:[]` (empty array). Assert error code -32602. Send `session/prompt` with `prompt:"plain string"` (not an array). Assert error code -32602.
- [AC-S1-17] The `recursive acp` subcommand is a registered CLI subcommand visible in `recursive --help` output.
  - how: Run `recursive --help` and assert the output contains the string `acp` in the subcommand listing. Run `recursive acp --help` and assert it prints usage information for the ACP subcommand (exit code 0).
