# Recursive ACP Protocol — Universal Editor Agent

Transform Recursive from a standalone coding agent into a universal editor-integrated agent via ACP v1. Any editor (Zed, JetBrains, VS Code via plugin) can embed Recursive as its native coding agent through stdio JSON-RPC. The agent gains bidirectional filesystem awareness — reading the editor's unsaved buffers while enforcing sandbox — and becomes interruptible mid-thought via collaborative cancellation that propagates from UI click to LLM stream drop to clean transcript pairing. Session persistence with history replay and cross-session MCP server lifecycle management makes it a first-class coding partner, not a one-shot tool.

## Sprints
### 1. Sprint 1 — Foundation: stdio handshake + text-only agent loop
- As a Zed user, I can configure my editor to use `recursive acp` as my coding agent, and the editor successfully completes the ACP initialize handshake, seeing Recursive's full capabilities declaration so it knows what tools and features are available.
- As a developer, I can create a new agent session via `session/new` with a working directory that becomes the sandbox root, and the session receives a stable, deterministic session ID I can use to reconnect later.
- As a developer, I can send a plain-text prompt via `session/prompt` and receive a streaming response as `agent_message_chunk` notifications, with the final chunk carrying a `stopReason` that clearly indicates whether the agent completed, hit a limit, or was interrupted.
- As a developer testing the ACP server, I can pipe a JSON-RPC initialize request into stdin and get a valid response on stdout, with no protocol-invalid bytes leaking to stdout (all logs go to stderr).

Key design decisions: (1) Copy McpServerRunner's proven stdin/stdout loop pattern — line-delimited JSON-RPC 2.0, UTF-8, one request at a time. (2) session/new maps cwd → resolve_within sandbox root directly — no impedance mismatch. (3) messageId generation uses content hash (SHA-256 of serialized content block) for stable, deterministic IDs that survive reconnect. (4) AgentRuntime::run() is the engine — ACP adapter is a thin EventSink→notification translator, never touching run_inner. (5) session/prompt is text-only in this sprint; tool_call notifications come in Sprint 2. Test: `echo '{"jsonrpc":"2.0","id":1,"method":"initialize",...}' | recursive acp` produces valid initialize result.

### 2. Sprint 2 — Interaction: tool call streaming + collaborative cancellation + permission bridge
- As a developer watching my agent work, I see live tool call notifications in my editor — when the agent invokes Bash, I see it appear as pending, then in_progress, then completed with output, each with the tool kind (execute/edit/read/search) and relevant file locations extracted from parameters.
- As a developer who accidentally sent the wrong prompt or sees the agent heading down a wrong path, I can hit cancel in my editor and the agent stops immediately — the LLM stream is dropped at the TCP level, the agent returns `stopReason: 'cancelled'` (not an error), and the transcript preserves clean tool-call↔tool-result pairing so nothing corrupts.
- As a security-conscious developer, when the agent needs permission to run a command or edit a file, my editor shows a permission dialog. I can approve or deny, and the agent respects my decision in real-time — denied operations produce clean error tool results rather than crashing the session.
- As a developer whose agent is waiting on a slow filesystem operation from the editor, I'm not stuck forever — any in-flight editor→agent fs/* RPC times out after 30 seconds and the agent gracefully degrades to local filesystem access or reports the timeout.
- As a tool author extending Recursive, I can add a new tool and give it a `kind()` that automatically maps to the right ACP tool_kind without touching any ACP code — the bridge is generic over the Tool trait.

This is the most architecturally invasive sprint. Three concurrent concerns: (1) LLM stream abort via tokio::select! in parse_sse_stream (both openai.rs and anthropic.rs) — the CancellationToken is checked between SSE chunks, and reqwest::Response is dropped on cancel, which closes the TCP connection. run_inner already maps Cancelled→FinishReason::Cancelled, so the ACP bridge just maps that to stopReason: 'cancelled'. (2) PermissionHook bridged to session/request_permission — the ACP server sends a request, blocks on the response channel, maps outcome→PermissionDecision. (3) ToolCard kind() default method added to the Tool trait with per-tool overrides. Locations extracted from tool args' path/cwd fields. The Transcript invariant (#8) is protected by: on cancel, every in-flight tool_call gets a synthetic error tool_result written before the agent turn ends, so pairing always holds.

### 3. Sprint 3 — Persistence & Editor Integration: session history, resume, editor filesystem, MCP multi-transport
- As a developer who closed my editor at the end of the day, when I reopen my project and reconnect to the agent, I see my entire conversation history replayed — every user message, agent response, and tool call appears in my editor exactly as it was, with stable message IDs that survive the disconnect.
- As a developer resuming work, I can use `session/resume` to reconnect to an existing session without seeing history replayed, and the agent immediately picks up where we left off — including reconnecting to all configured MCP servers with fresh subprocesses (old ones killed cleanly).
- As a developer editing files, the agent reads my unsaved editor buffer instead of the stale disk version — when the editor declares `fs.readTextFile=true`, the agent fetches content from the editor via `fs/read_text_file` RPC, and only falls back to local disk read if the editor doesn't have the file. Every path still goes through `resolve_within` so sandbox is never bypassed.
- As a developer with project-specific MCP servers, I can configure them per session (via `session/new`'s `mcpServers`), and they coexist with my global MCP config — session-scoped servers take priority on name conflicts, and all stdio subprocesses are killed when the session closes so nothing leaks.
- As a developer using a remote MCP server over HTTP or SSE, I can configure it as easily as a local stdio one — the ACP bridge handles all three transports transparently, and my agent tools work the same regardless of where the MCP server lives.

Three major subsystems converge: (1) SessionStore plumbing — session/load reads transcript, maps each entry to the correct session/update notification type (user_message_chunk, agent_message_chunk, tool_call with pending/in_progress/completed status), assigns stable messageId from content hash, streams them sequentially, then returns result. (2) Editor filesystem bridge — two new ACP-specific Tool implementations (ClientReadFile, ClientWriteFile) that issue fs/* RPCs to the editor. The RPC layer wraps CancellationToken + tokio::time::timeout(30s). On RPC failure or editor denial, ClientReadFile degrades to local Read; ClientWriteFile returns an error tool result. resolve_within always runs, even on editor-provided paths. (3) MCP bridge expansion — src/acp/mcp_bridge.rs converts session/new's mcpServers config into McpClientConfig, with transport discrimination (stdio→spawn subprocess, http/sse→connect URL). Session-scoped registry is separate from global config; merge logic is session-first on name conflict. session/close and session/load both iterate the registry, kill all stdio subprocesses, clear the map.

### 4. Sprint 4 — Ship: CLI integration, E2E pipeline, invariant enforcement, Zed dogfooding
- As a Recursive user, I can run `recursive acp` just like I run `recursive mcp` or `recursive http` — the subcommand appears in `--help`, accepts the same global flags, and feels native to the CLI.
- As a Recursive maintainer, I have an automated E2E test that simulates a full ACP client lifecycle — initialize, session/new, session/prompt (with tool calls), session/cancel, session/load, session/close — and asserts the exact notification sequence, cancel behavior, and clean shutdown with no leaked subprocesses.
- As a Recursive maintainer, I have automated invariant tests that run on every CI build: (a) ACP code never adds branches to run_inner, (b) every ACP filesystem path goes through resolve_within, (c) cancelled sessions preserve tool-call↔tool-result pairing in the transcript, (d) the LLM streaming cancel path actually drops the reqwest::Response.
- As a developer using Zed with the ACP plugin, I can add `recursive acp` as my agent, open a project, and have a complete coding session — ask a question, see the agent use tools, cancel mid-execution, close and reopen, see my history — all without touching a terminal.

This sprint is about hardening and delivery. (1) CLI: add Acp variant to the Cli enum in crates/recursive-cli/src/main.rs, wire to AcpServerRunner::run(). (2) E2E: follows CLAUDE.md e2e rules — new test suite under e2e/tests/ that starts `recursive acp`, sends JSON-RPC over stdin, reads stdout, asserts notification ordering. Must handle port allocation (use port 9095 or next available). (3) Invariants tests: AST-based check that run_inner doesn't import acp (same pattern as tests/invariants/loop_size_orthogonality.rs), grep-based check that all fs operations in acp/ path through resolve_within, transcript integrity check that iterates session transcript after cancel and verifies every Role::Assistant with tool_calls has adjacent Role::Tool messages. (4) Dogfooding: the real acceptance test is connecting Zed to Recursive ACP and having a real coding session — this validates all the protocol edges that unit tests miss (timing, chunk boundaries, editor-specific behavior).
