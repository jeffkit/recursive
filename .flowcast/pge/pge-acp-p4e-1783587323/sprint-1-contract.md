# Contract: Sprint 1 — Cancel Coordination & Permission Bridge (P4) (rev 2)

Implement immediate LLM stream cancellation via CancellationToken wired into parse_sse_stream with tokio::select!, ensuring the TCP connection drops via RST within 500ms (best-effort) and the transcript preserves all tool-call↔tool-result pairings (Invariant #8) by synthesizing ToolResults for in-flight tools. Build the permission bridge with per-tool session/request_permission notifications, 500ms debouncing with deduplication, and deterministic PermissionOutcome→PermissionDecision mapping with E2E verification of Allowed/Denied execution outcomes. Add 30s timeout for agent→client filesystem RPCs (defined as const `ACP_CLIENT_RPC_TIMEOUT_MS` with env-override). Document cooperative cancel semantics at src/acp/server.rs top.

## Criteria
- [S1-C1] A running prompt can be cancelled immediately via session/cancel and the run terminates with finishReason "cancelled" before natural completion.
  - how: E2E test: send session/prompt with a long instruction (e.g. "count to 100 slowly"), immediately send session/cancel. Assert the final end_turn notification contains stopReason:"cancelled" and the JSON-RPC response is a success (not error). Assert total elapsed time is <2s (far less than a full count-to-100 would take).
- [S1-C2] CancellationToken is wired into parse_sse_stream via tokio::select! so the stream aborts within one chunk cycle, not waiting for the next SSE event.
  - how: Unit test: create a CancellationToken and a slow SSE source (1 chunk/sec). Call parse_sse_stream with the token, cancel after 10ms. Assert the function returns within 100ms (one chunk cycle) without reading the next chunk. Use tokio::time::timeout to measure.
- [S1-C3] Cancelling the HTTP response triggers a TCP RST (hard close, not FIN) so the server detects the connection drop within 500ms (best-effort; no hard real-time guarantee).
  - how: Integration test: start a slow HTTP server that sends 1 byte/sec. Make a reqwest GET, cancel via CancellationToken then drop the Response. Confirm the server's read loop detects an error or EOF (RST semantics: no waiting for OS buffer drain). Measure elapsed time from drop to server EOF detection — assert <500ms but document as best-effort (CI may be slower). Use reqwest::Client::close() or Response::error_for_status combined with explicit Response drop, not implicit drop alone. Document that 500ms is advisory, not a hard real-time guarantee.
- [S1-C4] After cancel, the transcript has no orphan Role::Assistant messages with pending tool_calls — any in-flight tool gets a synthesized ToolResult so Invariant #8 is preserved.
  - how: E2E test: trigger a cancel while exactly one tool call is in-flight (e.g. a slow shell command). After cancel, dump the transcript and assert every Role::Tool message is immediately preceded by a Role::Assistant whose tool_calls list contains the matching id. Assert no Assistant message with non-empty tool_calls has no following Tool result.
- [S1-C5] session/request_permission notifications are emitted for every tool or shell action requiring approval, with structured context (tool name, path, command).
  - how: E2E test with permission="on": send session/prompt that triggers 3 different tool calls (Read, Write, Bash). Assert the client receives exactly 3 session/request_permission notifications, each containing a params object with fields: toolName (string), path (string or null), command (string or null).
- [S1-C6] Permission requests are debounced with a 500ms window: multiple tool calls arriving in the same Assistant message's tool_calls array (simultaneous, not sequential) consolidate into a single consolidated permission prompt with counts.
  - how: E2E test: send session/prompt that triggers 3 Read calls in a single Assistant message (all 3 in one tool_calls array, arriving at the debouncer on the same tick). Assert the client receives exactly 1 session/request_permission notification containing a consolidated object with count:3 and an array of the individual tool contexts. Assert no other request_permission notification arrives within the next 1s.
- [S1-C7] Deduplication during debouncing: identical permission requests (same tool name + same path) are merged into a single entry with aggregated count.
  - how: Unit test: feed the debouncer 3 identical Read requests and 2 different Write requests all within 500ms. Assert the consolidated prompt has 2 entries: Read with count=3 and Write with count=2, not 5 separate entries.
- [S1-C8] PermissionOutcome responses translate deterministically to PermissionDecision: Allowed → the agent proceeds with the tool call, Denied → the agent aborts without running the tool, with no ambiguity.
  - how: Unit test: enumerate every variant of PermissionOutcome (Allowed, Denied, Timeout) and feed each into the translation function. Assert Allowed maps to a PermissionDecision that lets the tool proceed. Assert Denied maps to a PermissionDecision that aborts. Assert Timeout maps to PermissionDecision::Denied (or a defined fallback). Test that unrecognized values produce a compile-time or match exhaustiveness error.
- [S1-C9] In-flight agent→client filesystem RPCs have a CancellationToken with 30s timeout, defined as the constant `ACP_CLIENT_RPC_TIMEOUT_MS` in src/acp/server.rs (overrideable via environment variable ACP_CLIENT_RPC_TIMEOUT_MS, defaults to 30000), so a hanging RPC does not block the session forever.
  - how: Unit test: invoke an RPC handler that sleeps indefinitely (no response). Assert that after 30s the call is cancelled and returns an error containing "timeout" or "cancelled". Assert the CancellationToken is properly wired as passed-down from the session lifecycle. Verify the constant is defined as a `const` or `lazy_static` with an env-override comment.
- [S1-C10] Cooperative cancel semantics are documented in a doc comment block at the top of src/acp/server.rs, explaining how CancellationToken flows through parse_sse_stream and what happens to in-flight tool calls on cancel.
  - how: Read src/acp/server.rs and assert the first non-comment, non-attribute line is preceded by a doc comment block (/// or /*! */) that contains at least the words "cancel", "CancellationToken", and "in-flight tool" or equivalent.
- [S1-C11] Client permission decisions (Allowed/Denied) are sent back to the server, and the server correctly executes or skips the tool accordingly.
  - how: E2E test: send session/prompt with permission="on" that triggers one tool call. Mock the permission response: (a) respond Allowed → assert the tool executes and its result appears in the transcript; (b) respond Denied → assert the tool is skipped (no tool result for that call) and the run continues with a summary or the next turn. Use two separate test cases or a sequential scenario.
- [S1-C12] session/cancel follows a defined JSON-RPC schema: request format is `{"method":"session/cancel","params":{}}`; success response is `{"result":true}`; invalid session ID or non-running session returns JSON-RPC error code -32000.
  - how: Unit/integration tests: (a) send a well-formed session/cancel to a running session → assert response `{"result":true}`. (b) send session/cancel with a non-existent session ID → assert JSON-RPC error response with code -32000. (c) send session/cancel to a session that is not in a running state → assert JSON-RPC error response with code -32000.
- [S1-C13] CancellationToken is wrapped in Arc before being shared across tasks: the cancel handler (signal receiver) and parse_sse_stream (read loop) each hold an Arc clone, never a bare reference.
  - how: Verify by inspection or static analysis: trace the CancellationToken from its creation point (in the HttpHandler or session lifecycle) to both (1) the signal handler that triggers cancel and (2) the read loop in parse_sse_stream. Assert at least one `Arc::clone()` call exists between them. A compile-time test asserting the type is `Arc<CancellationToken>` in both task bounds is acceptable.
- [S1-C14] Edge cases for session/cancel are handled correctly: (a) cancel on a session with no active run returns error -32000, not a panic; (b) consecutive identical cancels return success on first and either success or no-op on second (never double-trigger cancel logic); (c) a race between cancel and natural run completion produces a single deterministic finishReason (either "cancelled" or the natural reason, not both).
  - how: Three test cases: (a) send session/cancel to a session that was created but never started → assert JSON-RPC error -32000. (b) send session/cancel twice in quick succession → assert first returns result:true, second returns either result:true or an error indicating no-op (not a panic or double-cancel side effect). (c) send cancel simultaneously with the prompt completing naturally (use tokio::join! or a controlled timing setup) → assert the final end_turn notification contains exactly one stopReason, either "cancelled" or the natural finish reason.
