# Recursive ACP Protocol Support ‚Äî Remaining Sprints

Complete ACP v1 server support across 3 sprints (P5‚ÄìP7). The foundation (P0‚ÄìP4) is already committed. Sprint 1 delivers session persistence (load/resume) and the permission bridge. Sprint 2 adds editor-side filesystem tools and multi-transport MCP support with lifecycle cleanup. Sprint 3 hardens everything with CLI integration, E2E tests, invariants checks, and a Zed manual acceptance gate. Each sprint weaves AI into the UX: smart session summarization on load, read-ahead caching for editor fs, compaction hints for resume, and AI-generated test scenarios for E2E.

## Sprints
### 1. Sprint 1 ‚Äî Session Persistence & Permission Bridge
- As a developer resuming a long-running coding session, I want `session/load` to replay the full conversation history (user messages, agent messages, tool calls/results) via `session/update` notifications so the client sees a complete, accurate transcript without re-executing anything. Message IDs are derived from content hash per Decision #2.
- As a developer reconnecting after a network interruption, I want `session/resume` to restore the agent's context (system prompt, conversation history, MCP server connections) without replaying every notification, so I can continue from where I left off with minimal overhead.
- As a developer resuming a session, I want AI-summarized session context injected into the system prompt on load ‚Äî a concise paragraph describing what was accomplished, what was in-flight, and what blockers existed ‚Äî so I can reorient without manually scanning the transcript.
- As a developer, I want the agent to suggest smart compaction hints on resume ‚Äî identifying which early turns are safe to compress into a summary and which must remain verbatim ‚Äî so the context window stays under budget without losing critical details.
- As the session runtime, I want `session/load` and `session/resume` to tear down old stdio MCP subprocesses and spin up fresh ones from the provided `mcpServers` config, so stale connections never leak or cause port conflicts (Decision #5.4).
- As an agent developer, I want the PermissionHook to bridge `session/request_permission` ‚Üí `PermissionOutcome` ‚Üí `PermissionDecision` so that client-side permission dialogs (e.g., approve/reject a file write) work without blocking the event loop.
- As an agent developer, I want `session/cancel` notification to flow through `AgentRuntime::set_interrupt_token` and abort the LLM stream immediately via the `tokio::select!` on cancel_token already wired in P4b, with `stopReason: 'cancelled'` and no broken tool-call‚Üîtool-result pairing.
- As the agent runtime, I want all in-flight agent‚Üíclient fs/* RPCs to bind both a CancellationToken and a 30-second timeout (Decision #4a), so a hung client never wedges the agent forever.
- As a future maintainer, I want `src/acp/server.rs` to open with a 'Collaborative Cancellation' design doc section explaining the semantic contract (Decision #4c), so the cancel flow remains understandable and safely modifiable.

Kickoff from P5 (load/resume) plus all P4 leftover work (permission bridge). The permission bridge is the trickiest item because it spans ACP session ‚Üî Runtime ‚Üî PermissionHook ‚Äî must ensure the Runtime doesn't block waiting for a permission response forever. Use a oneshot channel per permission request with the same 30s timeout as fs/* RPCs. For smart summarization on load, implement a lightweight `summarize_transcript()` in `src/acp/session.rs` that runs as a fast local Lua/token heuristic (no LLM call ‚Äî too expensive and time-consuming on load); the summary can be enhanced later. Resume compaction hints are declarative metadata on each turn (compressible: bool) computed during session save.

### 2. Sprint 2 ‚Äî Editor Filesystem & Multi-Transport MCP
- As an ACP client (Zed/JetBrains), I want the agent to request file content from my editor buffer via `session/request_fs_read` when the client declares `fs.readTextFile=true`, so unsaved editor content is accessible without a disk round-trip. When the client does NOT declare the capability, the agent degrades gracefully to local `Read` tool (Decision #1).
- As the sandbox enforcer, I want ALL editor-filesystem operations ‚Äî both `ClientReadFile` and `ClientWriteFile` ‚Äî to pass through `tools::resolve_within` against the session's `cwd` (the sandbox root), so a malicious or misconfigured client cannot escape the workspace directory.
- As a power user, I want the agent to automatically detect when I'm editing a file and proactively prefetch the next logically related file (e.g., the corresponding test file or import) via read-ahead hinting, so tool call latency for the next Read feels instant.
- As an agent developer, I want diff-aware writes in `ClientWriteFile`: when both the old editor-buffer content and the new content are known, the agent sends only the diff (or a unified patch) to the client with a hint to apply it, reducing bandwidth and letting the editor preserve undo history.
- As an agent operator, I want the MCP bridge to support three transports (Decision #5.1): stdio (local subprocess), HTTP (remote server URL), and SSE (streaming endpoint). `mcpCapabilities` declares `{ http: true, sse: true }`. Stdio servers are spawned as child processes; HTTP/SSE servers are connected via reqwest without spawning anything.
- As an agent operator, I want session-scoped MCP server registries that merge with the global MCP config at `~/.recursive/mcp.json`. When a session provides `mcpServers` with a name that collides with the global config, the session-scoped entry wins. This registry lifecycle is per-session and cleaned up on `session/close` (Decision #5.2).
- As the session lifecycle manager, I want `session/close` to kill all stdio MCP subprocesses owned by that session (SIGTERM ‚Üí 5s grace ‚Üí SIGKILL), so no orphan processes linger. Load/resume also triggers this teardown before starting fresh servers (Decision #5.3).
- As a developer, I want AI-assisted MCP server auto-discovery: the agent scans `~/.recursive/mcp.json`, the session's `mcpServers` config, and any `mcp_servers` key in the project root's `.recursive.json`, merging them with conflict resolution warnings, so I don't have to manually register every MCP tool I need.

This sprint carries the most integration surface area. The read-ahead prefetch and diff-aware writes are ambitious additions beyond the literal requirement ‚Äî they leverage AI to anticipate developer intent and reduce friction. Auto-discovery of MCP servers from multiple config sources (user-global, project-local, session-scoped) is a force multiplier: a session might inherit 3 tools from the user config, 5 from the project, and 2 from its own mcpServers, all available transparently. Subprocess lifecycle must be bulletproof ‚Äî consider using a process group (setsid) so killing the group cleans up grandchild processes the MCP server itself spawned.

### 3. Sprint 3 ‚Äî CLI Integration, E2E & Invariants
- As a terminal user, I want `recursive acp` to appear alongside the existing `recursive mcp` and `recursive http` subcommands in `crates/recursive-cli/src/main.rs`, accepting the same `--provider` / `--model` / `--max-steps` flags, so I can start an ACP server with `recursive acp --provider openai`.
- As an E2E test suite, I want a scripted ACP client (a Rust or Python test harness) that sends a complete flow ‚Äî `initialize` ‚Üí `session/new` ‚Üí `session/prompt` (text) ‚Üí `session/prompt` (tool-using) ‚Üí `session/cancel` ‚Üí `session/load` ‚Üí `session/close` ‚Äî validating the full notification sequence, so every protocol state transition is covered by CI.
- As an invariants enforcer, I want an AST-level test (`tests/invariants/loop_size_orthogonality.rs`) that verifies no ACP code branches into `src/run_core.rs::run_inner`, so Invariant #1 is enforced by CI, not by code review.
- As an invariants enforcer, I want a runtime test that exercises ACP host-fs operations through `resolve_within` and asserts sandbox boundary rejection (escape attempt ‚Üí error), so Invariant #3 holds even for the editor-fs path.
- As an invariants enforcer, I want a test that cancels a mid-flight request and then inspects the transcript: every `Role::Tool` must have a paired `Role::ToolResult` (even if the result is a cancellation error), so Invariant #8 is never violated.
- As a release manager, I want `cargo test --workspace`, `cargo clippy --all-targets --all-features -- -D warnings`, and `cargo fmt --all -- --check` all green on the worktree before any merge, with no new `unwrap()`/`expect()` in non-test code, so the quality gates are airtight.
- As a Zed user, I want to manually configure my Zed ACP settings to point to `recursive acp` and run a real coding session (open a project, ask the agent to refactor a function, verify the file edit lands), so the integration is validated on an actual editor before shipping.
- As a future agent, I want the E2E test suite to include an AI-generated 'fuzz' scenario where the client sends malformed JSON, out-of-order messages, and duplicate session IDs, so the server gracefully rejects or handles each without crashing or leaking resources.
- As a developer on the project, I want a `.dev/journal/manual-<date>-acp-protocol.md` entry that records all non-obvious design decisions, gotchas encountered during these three sprints, and the Zed manual test results, so the institutional knowledge isn't lost.

The AI-generated fuzz scenario is an ambitious addition: instead of hand-crafting edge cases, the E2E suite should include a generative test that asks an LLM to produce 10-20 protocol-violating inputs and asserts the server recovers gracefully from each. The AST invariants test is critical ‚Äî the biggest risk across all sprints is someone accidentally adding a special case in run_inner for ACP. The Zed manual acceptance test should be scripted with explicit steps and expected outcomes, not 'works for me'.
