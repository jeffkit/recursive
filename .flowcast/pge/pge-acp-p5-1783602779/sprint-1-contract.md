# Contract: Sprint 1 — Session Persistence & Permission Bridge (rev 3)

Deliver session load/resume with full transcript replay, AI-summarized context injection, compaction hints, MCP lifecycle teardown, permission bridge via oneshot channel with 30s timeout, permission debouncing/dedup, session/cancel flowing through interrupt_token, and 30s timeout on all fs/* RPCs. Include a design doc section on collaborative cancellation. No LLM call on load; summarization uses fast local heuristic. The cancel_token field in AcpSession must be `Arc<CancellationToken>`. All types (PermissionOutcome, PermissionDecision) must be defined with exhaustiveness-tested translation.

## Criteria

- [ACP-S1-01] session/load replays full conversation history via session/update notifications without re-executing tools
  - how: Start a session, send 3 messages (including one that triggers a tool call with result). Close client. Send session/load with the saved session ID. Assert: each original user message and assistant message (including tool_calls and tool_results) is replayed as a session/update notification in correct order. Assert: no tool is re-executed (monitor tool invocation counter). Assert: final notification is session/loaded.

- [ACP-S1-02] Message IDs are derived from content hash per Decision #2
  - how: Inspect the saved transcript JSONL for a session with known messages. Assert: every message object has an `id` field matching SHA256(content). Assert: two identical user messages produce identical IDs. Assert: two different messages produce different IDs.

- [ACP-S1-03] session/resume restores agent context (system prompt, conversation history, MCP connections) without replaying every notification
  - how: Start a session with a custom system_prompt and two MCP server definitions in `mcpServers` (matching the ACP protocol spec field casing). Send 3 messages. Call session/resume with that session ID. Assert: the returned context includes the original system_prompt. Assert: the conversation history is present (check last message content matches). Assert: the MCP servers are re-initialized (check tool list includes tools from both servers). Assert: no session/update notifications are emitted during resume.

- [ACP-S1-04] AI-summarized session context is injected into system prompt on session/load
  - how: Create a session with at least 5 messages spanning multiple tool calls and a user-visible accomplishment. Save and close. Send session/load. Inspect the system_prompt in the session/loaded response. Assert: it contains a paragraph describing what was accomplished, what was in-flight, and blockers. Assert: the paragraph is generated by a local heuristic (not LLM) — verify by asserting zero LLM API calls were made during load via a provider mock that records all invocation attempts.

- [ACP-S1-05] Compaction hints (compressible: bool) are attached to each saved turn and surfaced on resume, governed by a configurable proximity heuristic
  - how: Run a session with 10 turns. Save session. Open the saved session file. Assert: each turn record has a `compressible` boolean field. Send session/resume. Assert: the response includes a `compaction_hints` array identifying which turn indices are safe to compress. Assert: turns beyond a recency threshold (e.g., all turns older than total_turns - 2) are marked compressible: true; the most recent turns within the threshold are compressible: false. Assert: the threshold is configurable — verify by setting a different threshold and repeating the check.

- [ACP-S1-06] session/load and session/resume tear down old stdio MCP subprocesses and spin up fresh ones from provided mcpServers config
  - how: Start a session with an MCP server (e.g., a test server that writes its PID to a known file). Save session. Call session/load with the same session ID but no mcpServers. Assert: the original MCP subprocess is killed (PID file no longer running / SIGCHLD observed). Call session/resume with a DIFFERENT mcpServers config. Assert: old MCP processes are killed; new processes matching the new config are spawned.

- [ACP-S1-07] PermissionHook bridges session/request_permission → PermissionOutcome → PermissionDecision without blocking the event loop for other concurrent requests
  - how:
    a) **Type definitions**: Define `PermissionOutcome` enum (Allowed | Denied | Timeout) and `PermissionDecision` enum (Allow | Deny). Implement a `translate(outcome: PermissionOutcome) -> PermissionDecision` function. Add an exhaustiveness unit test that instantiates every variant of PermissionOutcome and asserts the corresponding PermissionDecision is returned. Assert: any new variant added to PermissionOutcome causes a compile error in translate.
    b) **PermissionBridge struct**: Implement a PermissionBridge struct that owns a oneshot channel for each in-flight permission request. The bridge is created per `session/prompt` turn and dropped when the turn ends.
    c) **request_permission emission**: Before dispatching a tool call, the runtime emits a `session/request_permission` notification containing `{ toolName: string, path?: string, command?: string, permissionId: string }` to the client. This notification is non-blocking (fire-and-forget to the event loop).
    d) **Non-blocking event loop**: Configure a PermissionHook that simulates a client permission dialog. Send a session/prompt that triggers a 'write_file' tool call. Assert: a session/request_permission notification is emitted (capture via mock client). Assert: the runtime does NOT block the event loop for other concurrent requests — the mock client can respond asynchronously within 100ms. Assert: the tool call completes successfully after Allowed response.
    e) **Rejection path**: Re-test with PermissionDecision::Deny. Assert: the tool call is skipped with an appropriate error message in the transcript. Assert: the turn continues processing the next tool call (if any) without aborting.
    f) **Timeout**: When the client never responds, the oneshot channel times out after 30 seconds and PermissionDecision defaults to Deny (see S1-08).

- [ACP-S1-08] Permission requests use a oneshot channel with a 30-second timeout; timeout results in PermissionDecision::Deny
  - how: Send a session/prompt that triggers a tool call requiring permission. Configure the client to NEVER respond to request_permission. Assert: after 30 seconds, the tool call resolves with PermissionDecision::Deny (timeout). Assert: the runtime continues processing subsequent messages. Assert: no resource leak (check channel count after test — all oneshot senders/receivers are dropped).

- [ACP-S1-09] session/cancel notification aborts the LLM stream immediately via interrupt_token with stopReason: 'cancelled'
  - how:
    a) **E2E test**: Write a full E2E YAML test in `e2e/tests/` that performs: `session/new` → `session/prompt` with a long-running instruction (provider configured with ≥5s artificial delay) → `session/cancel` within 500ms. Assert: the prompt response arrives with `stopReason: 'cancelled'`. Assert: total elapsed time is **less than 2 seconds** (verifying cancellation is nearly instant, not waiting for the full generation). Assert: no tool is called as a side-effect of the partial generation.
    b) **parse_sse_stream unit test**: Add a unit test in `src/llm/` that: creates a `CancellationToken`, constructs an SSE source that emits 1 chunk per second, cancels the token 10ms later, and asserts `tokio::time::timeout(100ms, parse_sse_stream(...))` returns `Err(Error::Cancelled)` within 100ms.
    c) **TCP RST integration test**: Add an integration test that: starts an HTTP server sending 1 byte/sec, issues a reqwest GET, cancels the request (drops the Response), and verifies the server-side read loop detects an error or EOF within **500ms** of the client-side drop. This proves the `reqwest::Response` drop propagates reliably to the transport layer.

- [ACP-S1-10] Cancel does not break tool-call ↔ tool-result pairing (Invariant #8)
  - how: Start a fresh session (independent from S1-09) with a prompt designed to produce exactly 3 tool_calls in a single turn. The provider returns all 3 tool_calls at once. After the client has received and acknowledged the second tool_result notification, send session/cancel. Save the transcript. Then: dump the transcript and programmatically assert: every `Role::Tool` message has exactly one paired `Role::ToolResult` message (even if the result is a cancellation error). Assert: no orphaned `tool_call` or `tool_result` entries exist in the transcript (each tool_call in an assistant message has exactly one corresponding tool_result message, and vice versa). This must be verified in both an integration test and an E2E test.

- [ACP-S1-11] All in-flight agent→client fs/* RPCs bind both a CancellationToken and a 30-second timeout
  - how: Define a constant `ACP_CLIENT_RPC_TIMEOUT_MS: u64 = 30000` in `src/acp/server.rs`. Support override via the environment variable `ACP_CLIENT_RPC_TIMEOUT_MS`. Instrument the fs/* RPC dispatch to log timeout and token state. Send a `request_fs_read` to the client, and have the client NOT respond. Assert: after 30 seconds, the RPC resolves with a timeout error. Send a session/cancel while an fs/* RPC is in-flight. Assert: the RPC is aborted via CancellationToken before the 30s timeout. Assert: no hanging futures remain (check task count via tokio::spawn tracking).

- [ACP-S1-12] src/acp/server.rs opens with a 'Collaborative Cancellation' design doc section
  - how: Read the first 50 lines of src/acp/server.rs. Assert: the file begins with a doc comment block titled 'Collaborative Cancellation' or '## Collaborative Cancellation'. Assert: the section explains the semantic contract: cancel is a request (not a demand), the agent drains in-flight tool results before stopping, and stopReason: 'cancelled' is set on the last assistant message. Assert: the section references Decision #4c.

- [ACP-S1-13] session/load and session/resume handle error scenarios gracefully with standard JSON-RPC error responses
  - how: Test each of the following:
    a) Non-existent session ID: Call session/load with a non-existent session ID — assert the response contains a JSON-RPC error with code -32000 and a descriptive message.
    b) Corrupted session file: Manually corrupt a saved session JSONL file (e.g., truncate mid-JSON), then call session/load — assert a -32000 error with message indicating parse failure is returned; the server does not crash.
    c) Cancel idempotency: Send session/cancel for a prompt that has already completed — assert the cancel is idempotent (returns success without error).
    d) Invalid mcpServers: Configure mcpServers with an invalid path (e.g., a binary that does not exist), then call session/resume — assert the server logs a warning but still returns a valid context (the invalid server is gracefully skipped, not blocking the entire resume).
    e) Cancel non-running session: Create a session but never send a prompt (session is created but no turn is running). Send session/cancel. Assert: the response is a JSON-RPC error with code -32000 and a message indicating there is no active turn to cancel.
    f) Double cancel: Start a prompt, send session/cancel and assert the first cancel returns `result: true` (success). Send the same cancel again. Assert: the second cancel is a no-op — returns `result: true` without triggering any side effects (no double tool-result, no second stopReason emitted, no crash).
    g) Cancel + natural completion race: Use `tokio::join!` to send both a cancel and a prompt response simultaneously. Assert: `end_turn` emits exactly one stopReason (either 'cancelled' or 'endTurn'/'stop', never both). Assert: no duplicate session/update notification is emitted.

- [ACP-S1-14] Permission debouncing: multiple tool calls within the same `tool_calls` array are consolidated into a single session/request_permission notification
  - how: Implement a 500ms debounce window in the permission dispatch. When a single assistant turn produces N tool_calls, group them: emit one consolidated `session/request_permission` notification containing all N tool entries (as a `tools` array in the notification body), rather than N separate notifications. The debounce MUST NOT delay tool execution beyond 500ms (clients see the consolidated notification within 500ms of the first tool dispatch). Send a prompt that triggers 3 tool_calls at once. Assert: exactly 1 `session/request_permission` notification is emitted (not 3). Assert: the notification body contains a `tools` array with 3 entries, each with `toolName` and optional `path`/`command`. Assert: the client can respond once with an array of PermissionDecisions, one per tool. If the 500ms window expires or the next tool_call from a different turn arrives, the current batch is flushed immediately.

- [ACP-S1-15] Permission deduplication: identical tool+path pairs within the debounce window are aggregated
  - how: If the same tool name AND the same path appear multiple times within the same debounce window (500ms), consolidate them into a single entry with field `count: N`. Different tools or different paths remain as separate entries. Send a prompt that triggers: 2× `write_file` on the same path, 1× `write_file` on a different path, and 1× `read_file`. Assert: the consolidated notification contains exactly 3 entries: `{ toolName: "write_file", path: "/shared", count: 2 }`, `{ toolName: "write_file", path: "/other", count: 1 }`, `{ toolName: "read_file", count: 1 }`. Assert: the client can respond once per consolidated entry.

- [ACP-S1-16] AcpSession::cancel_token must use `Arc<CancellationToken>` to guarantee shared ownership across concurrent tasks
  - how: Change the `cancel_token` field type in `AcpSession` from bare `CancellationToken` to `Arc<CancellationToken>`. In `session/new` and `refresh_cancel_token`, wrap the token in `Arc::new(...)`. Add a compile-time test that asserts both `parse_sse_stream` and the cancel handler (e.g., `handle_session_cancel`) hold an `Arc<CancellationToken>` — this can be a type-assertion test that instantiates each function's signature with an explicit type annotation. Verify that dropping the `AcpSession` does not cancel the token if other `Arc` clones are held by in-flight tasks.
