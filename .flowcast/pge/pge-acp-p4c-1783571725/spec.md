# Recursive ACP Protocol Support — Agent Client Protocol v1 Integration

Make Recursive a first-class ACP server that any ACP-compliant editor (Zed, JetBrains, VS Code via plugin) can invoke as a coding agent over stdio JSON-RPC. The implementation spans 7 sprints (P1–P7, with P0 protocol types already shipped) and covers: full session lifecycle with streaming agent output, tool-call visibility with live status updates, collaborative cancellation that instantly aborts in-flight LLM streams, a permission bridge that lets editors gate dangerous operations, session history replay and resume, bidirectional editor filesystem access (agent reads unsaved buffers), multi-transport MCP server orchestration, and a polished `recursive acp` CLI. AI features are woven throughout: smart tool-kind inference for custom tools, adaptive permission learning that remembers user choices, intelligent context-window management for long sessions, and AI-driven MCP server discovery and ranking.

## Sprints
### 1. Sprint 1 — stdio JSON-RPC Loop + initialize Handshake
- As a Zed user, I can configure `recursive acp` as my coding agent and see it respond to an `initialize` request with protocol version, agent info, and full capability declaration so the editor knows what this agent supports.
- As a developer integrating Recursive, I get a clean stdio JSON-RPC loop that follows the same battle-tested patterns as the existing MCP server, with no interleaved output on stdout and all diagnostics on stderr.
- As a protocol implementor, I can trust that malformed JSON, unknown methods, and transport errors produce proper JSON-RPC error responses rather than crashing the server.
- As a QA engineer, I can run a single-shot ACP client script that pipes `initialize` to the agent process and asserts the response shape, including `protocolVersion: 1`, `agentInfo.name: "recursive"`, and the full `agentCapabilities` object with all future-phase capabilities already declared (even if unimplemented — this lets editors discover capabilities upfront without probing).

Foundation sprint. Reuse `src/mcp_server.rs::McpServerRunner::run` pattern directly: stdin BufReader line-by-line, stdout locked writer, JSON-RPC 2.0 dispatch table. The `agentCapabilities` object should declare ALL capabilities planned through Sprint 7 (`promptCapabilities`, `toolCallNotifications`, `loadSession`, `resume`, `fs.readTextFile`, `fs.writeTextFile`, `mcpCapabilities`, `terminalCapabilities: false`) so that capability negotiation works from day one. Stderr goes to `eprintln!` — the loop must NEVER write non-JSON to stdout. Add a 60-second idle timeout that shuts down cleanly with `ServerShutdown` reason.

### 2. Sprint 2 — Session Lifecycle + Text-Only Prompt Loop
- As a Zed user, I type a prompt in the editor's agent panel, hit send, and see the agent's response stream in as `agent_message_chunk` notifications in real time — just like chatting with any other coding agent.
- As an editor integrating Recursive, `session/new` gives me a stable session ID that I can use for subsequent prompts within the same conversation context, with the agent's working directory sandboxed to the `cwd` I provide.
- As a power user running long refactoring sessions, I can send multiple sequential `session/prompt` calls in the same session and the agent maintains context across turns (accumulated transcript, tool call history, working memory), ending each turn with a proper `end_turn` notification containing `stopReason`.
- As a protocol implementor, `session/prompt` accepts `ContentBlock[]` input — mixing `text`, `resource_link`, and `image` blocks — and the ACP bridge correctly converts them into Recursive's internal `Message` format, preserving relative ordering and metadata.
- As an AI-assisted developer, when I reference a file path or symbol in my prompt, the agent automatically infers context from my workspace structure before responding, making the interaction feel intelligent rather than blind.

This is the first sprint that connects ACP to the agent runtime. Key architectural decision: the ACP `EventSink` adapter wraps `AgentRuntime::event_sink()` and translates each `Event` into the appropriate `session/update` notification shape. The `session/new` handler creates an `AgentRuntime` instance with `cwd` mapped to the sandbox root via `resolve_within`. `messageId` generation uses content-hash (SHA-256 of serialized content block) for stable, deterministic IDs. AI feature: when `session/new` is called, the agent pre-indexes the workspace structure (directory tree + git status) so follow-up prompts benefit from ambient context without explicit file reads, making responses faster and more relevant.

### 3. Sprint 3 — Tool Call Visibility with Kind Classification & Live Status
- As a Zed user watching the agent work, I see each tool invocation appear in the editor panel as it happens — `tool_call` notification when the agent decides to act, showing the tool name, kind icon (📖 for Read, ✏️ for Edit, ▶️ for Bash, 🔍 for Glob), and target path.
- As a developer, I can follow tool execution through three live status transitions — `pending` (agent requested it), `in_progress` (tool is running), and `completed` (result returned) — so I'm never left wondering what the agent is doing.
- As a Recursive tool author, my custom tool automatically gets a reasonable `kind` label thanks to AI-powered inference that analyzes the tool name and parameter schema, falling back to `Other` only when genuinely ambiguous.
- As a protocol implementor, `tool_call_update` notifications carry structured `locations` data — file paths, line numbers, symbols — extracted from tool parameters so the editor can show inline annotations and open affected files.
- As a reviewer, when the agent runs a multi-step refactor (read → edit → test), I see each step's tool call with the output streaming in the `result` field of `tool_call_update`, giving me a live play-by-play of the agent's reasoning and actions.

The `Tool` trait gets a new default method `kind() -> ToolKind` with an enum: `Read`, `Write`, `Edit`, `Execute`, `Search`, `Other`. Each built-in tool overrides it explicitly. AI feature for custom tools: a static registry analyser that runs at `initialize` time, using heuristic matching (name + param names + description text) to classify un-annotated tools — e.g., a tool named `run_tests` with a `command` param is classified as `Execute`. The `Event` enum gets new variants or a mapping layer that translates `ToolCall`/`ToolResult` events into the three-phase notification sequence. The `locations` extractor uses a `Tool::locations(&self) -> Vec<Location>` default that returns empty, overridden by path-aware tools.

### 4. Sprint 4 — Cancellation, LLM Stream Abort & Permission Bridge
- As a Zed user who hit Escape during a slow agent response, the LLM stream stops immediately (not after the next chunk), the agent returns `stopReason: "cancelled"` instead of an error, and I can immediately send a new prompt without waiting for a timeout.
- As a developer, when the agent wants to run `rm -rf` or modify a file outside my workspace, I get a permission dialog in my editor, can inspect the full details, and choose Allow/Deny/AllowAlways — and the agent respects my choice.
- As a security-conscious user, I can trust that even if the agent's `fs/read_text_file` RPC to the editor hangs, a 30-second timeout kicks in and the agent falls back to local filesystem read rather than blocking forever.
- As a protocol implementor, when `session/cancel` arrives, all in-flight operations are notified: the LLM SSE loop `tokio::select!`s on the cancel token and drops the HTTP connection, pending `fs/*` RPCs are abandoned, and the next `session/update` notification carries `stopReason: "cancelled"` with a clean transcript where every `tool_call` still has its paired `tool_result`.
- As an AI-augmented developer, the permission bridge learns from my choices — if I consistently deny a specific tool on a specific path pattern, the agent starts auto-denying similar requests proactively, reducing interruption while maintaining safety.

Most complex sprint. Three subsystems change: (1) Provider layer: `parse_sse_stream` in both `src/llm/openai.rs` and `src/llm/anthropic.rs` gets a `CancellationToken` parameter and uses `tokio::select!` to race the next SSE chunk against token cancellation. On cancel, `reqwest::Response` is dropped (closing the TCP connection), and the function returns `Err(Error::Cancelled)`. `run_inner` already maps this to `FinishReason::Cancelled`. (2) Permission bridge: a new `AcpPermissionHook` struct implements `PermissionHook`, sending `session/request_permission` notifications and awaiting `PermissionOutcome` responses via a oneshot channel registry keyed by request ID. The AI learning feature: a simple frequency-based cache that tracks (tool, path_prefix) → (allow_count, deny_count) and auto-denies when deny_ratio > 0.8. (3) In-flight RPC guard: the `fs/*` RPC sender wraps with `tokio::time::timeout(30s)` and a cancel token. The `src/acp/server.rs` top-of-file documentation must include a "Collaborative Cancellation" section explaining that cancellation is best-effort cooperative — the agent stops work as fast as the current I/O lets it, but the LLM provider connection drop is the primary fast-path.

### 5. Sprint 5 — Session History Replay & Resume
- As a Zed user who closed the editor and reopened it the next morning, I run `session/load` and my entire conversation with the agent — every message, every tool call, every result — replays in the editor panel, letting me pick up exactly where I left off.
- As a developer resuming a session, `session/resume` restores the agent's full context (transcript + working memory + MCP server connections) without replaying the history visually, so I get instant context with no UI clutter.
- As a team lead reviewing an agent session, the replay uses stable `messageId` values (content-hash based) so the same historical message always gets the same ID — enabling reliable annotations, bookmarks, and cross-session references.
- As an AI-assisted developer, when I resume a long session after hours away, the agent automatically generates a concise summary of where we left off — what was accomplished, what's in progress, and what the next step should be — injected as a system message before my next prompt.
- As an MCP power user, `session/load` and `session/resume` correctly reconnect my MCP servers: old stdio subprocesses are killed, new ones are spawned with the configuration I provided, and http/sse connections are re-established — with zero leaked zombie processes.

`session/load` reads the transcript from `SessionStore`, iterates messages in order, and emits `session/update` notifications (`user_message_chunk`, `agent_message_chunk`, `tool_call`, `tool_call_update`) for each. After replay, returns `result: null`. `messageId` uses SHA-256 of the serialized message content — this is deterministic, collision-resistant, and requires no schema changes. `session/resume` restores `AgentRuntime` state without replay. AI feature: the auto-summary generator uses the agent's own LLM (via a lightweight system prompt) to produce a 3-5 bullet summary from the last N messages, stored in scratchpad and injected on next prompt. MCP reconnection: `session/load` and `session/resume` handlers receive `mcpServers` config, call `kill_all_mcp_servers()` on the old session, then `spawn_mcp_servers(new_config)` before returning.

### 6. Sprint 6 — Editor Filesystem Bridge & MCP Multi-Transport Orchestration
- As a Zed user with unsaved changes in my editor buffer, when the agent reads that file, it sees my latest unsaved content from the editor rather than the stale disk version — so refactoring suggestions account for my in-progress edits.
- As a developer, when the agent writes a new file or edits an existing one, the changes appear in my editor immediately as unsaved changes, and I can review them before committing to disk.
- As an MCP power user, I can connect local stdio-based MCP servers (the agent spawns them), remote HTTP MCP servers (the agent connects to a URL), and SSE-based MCP servers — all within the same session, with the agent orchestrating tool discovery across all of them.
- As a polyglot developer, when I declare `mcpServers` in my session config, the agent automatically merges their tool lists with Recursive's built-in tools, and if two servers expose a tool with the same name, the session-scoped server wins over the global config.
- As a system administrator, when my session ends (either `session/close` or editor exit), all stdio MCP child processes are killed, all HTTP connections are closed, and `ps` shows zero leaked subprocesses — confirmed by the cleanup invariant test.
- As an AI-augmented developer, the agent intelligently ranks MCP servers by relevance to each prompt — prioritizing the server whose tool descriptions best match my current task, and deprioritizing irrelevant servers to reduce tool-discovery noise.

Editor fs bridge: two new internal tool implementations — `ClientReadFile` and `ClientWriteFile` — that send `fs/read_text_file` and `fs/write_text_file` JSON-RPC requests to the client via the ACP transport. These are registered only when the client declares `fs.readTextFile: true` / `fs.writeTextFile: true` in its capabilities. The existing `Read` and `Edit`/`Write` tools gain a pre-check: if client fs is available, prefer client buffer; otherwise fall back to local. ALL paths still go through `resolve_within` — the editor fs is an additional data source, not a sandbox bypass. MCP multi-transport: `src/acp/mcp_bridge.rs` handles three transport types. For stdio, spawn a child process (like `src/mcp.rs` does); for http, use `reqwest` to the URL; for sse, use `reqwest` streaming. Tool lists are merged with session-overrides-global conflict resolution. The AI ranking feature: a TF-IDF similarity scorer that compares the current prompt text against each MCP server's `tools/list` descriptions, producing a relevance-ordered tool list for the LLM context — reducing token waste on irrelevant tools. Cleanup is enforced by a `Drop` impl on the session-scoped MCP registry that sends SIGTERM (then SIGKILL after 2s) to all child processes.

### 7. Sprint 7 — CLI Subcommand, End-to-End Validation & Invariant Gate
- As a developer, I run `recursive acp` in my terminal and the agent starts listening on stdio for ACP JSON-RPC messages, with the same CLI ergonomics as `recursive mcp` and `recursive http`.
- As a Zed user setting up the agent, I configure my editor with the command `recursive acp` and it Just Works — the agent connects, handshakes, and is ready for prompts with zero additional configuration.
- As a QA engineer, the E2E suite runs a scripted ACP client that exercises the full lifecycle: initialize → session/new → session/prompt (triggers tool call) → session/cancel → session/load (verify history) — and asserts every notification in the expected sequence with correct payloads.
- As a Recursive maintainer, three invariant tests gate every commit touching ACP code: (1) no ACP dispatch branches exist inside `run_inner` (AST static analysis), (2) every ACP filesystem path goes through `resolve_within` (grep assertion), and (3) after cancellation, every `tool_call` in the transcript has a paired `tool_result` (transcript integrity check).
- As the final acceptance test, I open Zed, point it at `recursive acp`, type "Refactor the user authentication module to use Argon2 instead of bcrypt" — and the agent reads relevant files, proposes changes, waits for my approval on destructive operations, and applies the refactor across multiple files, all visible as live tool calls in the editor panel.

Final integration sprint. CLI: add `Acp` variant to the `Cli` enum in `crates/recursive-cli/src/main.rs`, wired to `AcpServerRunner::run()`. The `--help` output must show `acp` alongside `mcp` and `http`. E2E tests follow the `CLAUDE.md` e2e rules: scripted ACP client binary that runs inside the `recursive-e2e` container, asserting notification sequences with `argusai`. The three invariant tests live in `tests/invariants/acp_loop_size.rs`, `tests/invariants/acp_sandbox.rs`, and `tests/invariants/acp_transcript_pairing.rs` — these are fast (no network, no containers) and run as part of `cargo test --workspace`. The final Zed manual acceptance test is not automated but serves as the ultimate integration check. After this sprint, Recursive is a production-grade ACP coding agent.
