# Contract: Sprint 2 — Session Lifecycle + Text-Only Prompt Loop (rev 2)

Connect ACP to the agent runtime: session/new creates a sandboxed AgentRuntime; session/prompt accepts text-only ContentBlock[] and streams agent_message_chunk + end_turn notifications via an EventSink adapter; sequential prompts maintain full conversation context; stopReason reports turn outcome; messageId uses SHA-256 hex digest (first 12 chars) for determinism.

## Criteria
- [AC-2.1] session/new returns a stable session ID valid for subsequent session/prompt calls, with cwd sandboxed via resolve_within
  - how: Send {"jsonrpc":"2.0","method":"session/new","params":{"cwd":"/workspace/project"},"id":1}. Assert response.result.sessionId is a non-empty string. Immediately send session/prompt with that sessionId; assert it succeeds (no 'session not found' error). Repeat with cwd containing path traversal (e.g., cwd='/workspace/../../../etc') and assert the sandbox rejects or normalizes it — the agent must not escape the workspace root.
- [AC-2.2] session/prompt with a text ContentBlock streams agent_message_chunk notifications in real time and terminates with an end_turn notification carrying stopReason
  - how: Send session/prompt with params.prompt=[{"type":"text","text":"Say hello and list the current directory"}]. Capture all JSON-RPC notifications on stdout until end_turn. Assert: at least one agent_message_chunk notification arrives with a non-empty content delta; the final notification is end_turn with a valid stopReason string (not null, not empty); no error response is received.
- [AC-2.3] session/prompt rejects or gracefully ignores non-text ContentBlock types (resource_link, image) — only text blocks are processed in Sprint 2; unsupported block types produce a parse error or are silently filtered
  - how: Send session/prompt with params.prompt=[{"type":"text","text":"hello"},{"type":"resource_link","uri":"file:///workspace/project/README.md"},{"type":"image","uri":"data:image/png;base64,iVBORw0KGgo="}]. Assert: the server either (a) returns a JSON-RPC error indicating unsupported block types, or (b) accepts the request but only processes the text block (agent response shows no awareness of README or image contents). In either case, the server must not crash, hang, or silently resolve the resource/image as if they were supported. A third case (c) sending only non-text blocks, e.g. [{"type":"resource_link","uri":"file:///workspace/README.md"}], must also produce an error or a graceful end_turn — not a crash.
- [AC-2.4] Multiple sequential session/prompt calls in the same session maintain context — the agent remembers prior turns' transcript, tool calls, and scratchpad state
  - how: Step 1: session/prompt with "My name is Alice and I'm working on a Rust project called stellar". Step 2: session/prompt with "What's my name and what project am I working on?" Assert: the agent's agent_message_chunk output for step 2 mentions 'Alice' and 'stellar', proving cross-turn context retention. Verify the end_turn notification for step 2 arrives with stopReason="end_turn" (not an error or timeout).
- [AC-2.5] Every completed session/prompt turn produces exactly one end_turn notification as the final event, and stopReason is one of the defined values (end_turn, max_turns, cancelled, error)
  - how: Run three distinct session/prompt calls: (a) a simple text prompt that completes normally, (b) a prompt that triggers a tool call (e.g., 'read README.md') and finishes, (c) a prompt that hits max_turns. For case (c), use a session/new configuration or mock that limits the agent to a single step (e.g., sending a prompt that demonstrably requires tool use but with maxSteps=1 set in the session config, or pre-populating the session with a step counter already at its limit). For each case, assert: exactly one end_turn notification is received; it is the last notification for that turn; stopReason is a non-empty string matching the expected outcome for that scenario ("end_turn" for a and b, "max_turns" for c).
- [AC-2.6] messageId generation is deterministic — same content block produces the same SHA-256-based messageId across sessions, using lowercase hex encoding of the first 12 characters of the digest
  - how: Open two separate sessions via session/new. In session A, send session/prompt with text 'Hello world'. Record the messageId from the session/update notification's user_message_chunk. In session B, send session/prompt with identical text 'Hello world'. Assert the two messageId values are byte-for-byte identical. Assert the messageId is exactly 12 characters long and contains only lowercase hexadecimal characters [0-9a-f] (e.g., 'a1b2c3d4e5f6'). Repeat with a different prompt to confirm the hash is content-sensitive, not a static counter.
- [AC-2.7] EventSink adapter translates agent_message_chunk and end_turn events into valid session/update JSON-RPC notifications with correct sessionId and fields
  - how: Trigger a prompt that causes the agent to respond (e.g., 'Say hello and explain what you can do'). Assert the notification stream contains: (1) one or more agent_message_chunk notifications, each carrying a non-empty content delta and the same sessionId as the active session; (2) a final end_turn notification with fields sessionId (matching), stopReason (one of: end_turn, max_turns, cancelled, error), and turnId (non-empty string). No other notification types (e.g., tool_call, tool_call_update) are emitted — Sprint 2 only surfaces these two notification kinds. Every notification must be a well-formed JSON-RPC 2.0 notification ("jsonrpc":"2.0", "method":"session/update").
- [AC-2.9] session/prompt with a stale or nonexistent sessionId returns a proper JSON-RPC error (not a crash or hang)
  - how: Send session/prompt with params.sessionId='nonexistent-session-id'. Assert: the JSON-RPC response has "error" (not "result"); error.code is a valid JSON-RPC error code (e.g., -32000 for application error); error.message is descriptive; the server remains responsive — a subsequent session/new call still succeeds.
- [AC-2.10] session/prompt handles an empty prompt array gracefully — either returns a sensible no-op or a prompt for input, but does not crash or hang
  - how: Send session/prompt with params.prompt=[]. Assert: the server returns either (a) an error response indicating empty prompt is invalid, or (b) an end_turn notification with a helpful agent message asking for input. In either case, the server must not hang, crash, or emit malformed JSON.
