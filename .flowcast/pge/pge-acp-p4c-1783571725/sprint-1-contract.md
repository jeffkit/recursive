# Contract: Sprint 1 — stdio JSON-RPC Loop + initialize Handshake

Foundation sprint establishing the ACP stdio transport, JSON-RPC 2.0 dispatch loop, initialize handshake with full forward capability declaration, error resilience, idle timeout, and automated QA verification.

## Criteria
- [S1-C1] initialize handshake returns correct response shape: `protocolVersion` is the integer 1, `agentInfo.name` is the string "recursive", and `agentCapabilities` is a non-empty object.
  - how: Pipe `{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":1,"clientInfo":{"name":"test","version":"0"},"clientCapabilities":{}}}` to stdin of `recursive acp`. Read one JSON line from stdout. Assert: `.jsonrpc == "2.0"`, `.id == 1`, `.result.protocolVersion == 1`, `.result.agentInfo.name == "recursive"`, `.result.agentCapabilities` is a non-null object with at least 5 keys.
- [S1-C2] Full forward capability declaration: `agentCapabilities` object declares all planned Sprint 2–7 capabilities — `promptCapabilities`, `toolCallNotifications`, `loadSession`, `resume`, `fs.readTextFile`, `fs.writeTextFile`, `mcpCapabilities`, `terminalCapabilities` (explicitly `false`) — so editors can discover capabilities upfront without probing.
  - how: In the same initialize response (S1-C1), assert all 8 keys exist in `.result.agentCapabilities`. Specific value checks: `.result.agentCapabilities.terminalCapabilities === false`. The remaining 7 keys must be truthy (non-null, non-false). This can be verified with a single `jq` expression against the captured response.
- [S1-C3] Stdio JSON-RPC loop reads newline-delimited JSON from stdin line-by-line, dispatches each request to a method handler, and writes exactly one JSON-RPC response line per request to stdout.
  - how: Pipe 3 sequential initialize requests with distinct `id` values (1, 2, 3) to `recursive acp` stdin. Capture stdout (3 lines). Assert each line is valid JSON, each has the matching `.id`, and all have `.jsonrpc == "2.0"`. No extra lines, no partial writes.
- [S1-C4] Clean transport separation: stdout carries ONLY valid JSON-RPC messages (one per line). All diagnostics, logs, startup messages, and error context go to stderr exclusively. No printf-debugging or welcome banners on stdout.
  - how: Start `recursive acp` and capture stdout and stderr separately (e.g., `2>stderr.log 1>stdout.log`). Send one initialize request. Assert every line in `stdout.log` parses as valid JSON with `.jsonrpc == "2.0"`. Assert `stderr.log` is non-empty and contains human-readable text (logs). Verify stdout has zero non-JSON bytes using `jq empty stdout.log` (must exit 0 for every line).
- [S1-C5] Malformed JSON input produces a JSON-RPC Parse error response (`error.code: -32700`, `error.message` describing parse failure) rather than crashing, hanging, or writing garbage.
  - how: Send literally `{bad json` (no closing brace, no quote closure) followed by newline to `recursive acp` stdin. Read one line from stdout. Assert: `.jsonrpc == "2.0"`, `.id == null` (parse errors have no id), `.error.code == -32700`, `.error.message` is a non-empty string. Process must still be alive and accept the next valid request — send a valid initialize and assert it succeeds.
- [S1-C6] Unknown method name produces a JSON-RPC Method not found error response (`error.code: -32601`) — the server must not crash, panic, or exit on unrecognized methods.
  - how: Send `{"jsonrpc":"2.0","id":99,"method":"nonexistent_method","params":{}}`. Read one line from stdout. Assert: `.id == 99`, `.error.code == -32601`, `.error.message` is a non-empty string. Then send a valid initialize (id: 100) and assert it still returns a successful result — proving the dispatch loop survived.
- [S1-C7] Invalid JSON-RPC request structure (missing required `jsonrpc` field or `method` field in a request) produces an Invalid Request error (`error.code: -32600`).
  - how: Send `{"id":1,"method":"initialize"}` (missing `jsonrpc`). Assert response has `.error.code == -32600`. Send `{"jsonrpc":"2.0","id":2}` (missing `method`). Assert response has `.error.code == -32600`. In both cases the process stays alive and the next valid request succeeds.
- [S1-C8] JSON-RPC notification (a message with no `id` field) is processed silently — no response is written to stdout. The server does not crash or produce a spurious response.
  - how: Send `{"jsonrpc":"2.0","method":"initialize","params":{...}}` (no `id`). Then immediately send a valid request with `id: 1`. Read one line from stdout. It must be the response to `id: 1` (not a response with `id: null`). There must be exactly one stdout line, proving the notification was swallowed cleanly.
- [S1-C9] 60-second idle timeout: if no input arrives on stdin for 60 seconds, the server shuts down cleanly — logs a message to stderr containing the reason `ServerShutdown`, and exits with code 0.
  - how: Start `recursive acp`, send one initialize request to confirm it's alive, then send no further input. Wait 62 seconds. Assert the process has exited (no longer running). Assert exit code is 0. Assert stderr contains the string `ServerShutdown`. This can be scripted with `timeout 65` wrapper and exit-code check.
- [S1-C10] The server process responds to SIGTERM gracefully: exits with code 0 (or 143 from the signal), and does not leave stale stdout in a broken state.
  - how: Start `recursive acp`, send one initialize request, then send SIGTERM. Assert process exits within 2 seconds. Check exit code is 0 or 143. Stdout from before the signal must still be valid JSON — no truncated writes.
- [S1-C11] The implementation reuses the `McpServerRunner::run` pattern: `BufReader::new(stdin()).lines()` for reading, a locked `stdout()` writer, and a dispatch table (`HashMap<&str, fn>`) mapping method names to handlers. The loop is a single `while let Ok(Some(line))` that never branches into agent runtime logic.
  - how: Code review gate: grep `src/acp/server.rs` (or whichever file hosts the loop) for `BufReader`, `stdin`, `lines()`, `stdout`, `lock`, and `HashMap`. Confirm the loop body is ≤ 30 lines and contains zero references to `AgentRuntime`, `run_inner`, `Provider`, or `Session`. The dispatch table must only have `"initialize"` mapped in Sprint 1.
- [S1-C12] QA automation: a single-shot shell script (living in `e2e/tests/`) pipes an initialize request to `recursive acp` and asserts the full response shape in one pipeline, exiting 0 on success. This script is runnable both inside the `recursive-e2e` container and locally with `cargo run -- acp`.
  - how: A script at `e2e/tests/01-acp-initialize.sh` (or `.yaml` for argusai) that: (1) starts `recursive acp`, (2) sends the initialize JSON, (3) captures the first stdout line, (4) runs `jq` assertions on `protocolVersion`, `agentInfo.name`, and all 8 `agentCapabilities` keys, (5) kills the process, (6) exits 0 only if all assertions pass. Must be listed in `e2e/e2e.yaml` and pass `argusai -c e2e.yaml`.
