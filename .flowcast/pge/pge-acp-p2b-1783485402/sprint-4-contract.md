# Contract: Sprint 4 — session/cancel + LLM stream abort + permission bridging (rev 2)

Most complex sprint. Critical path: (1) register session/cancel as an ACP method with proper validation, (2) materialize CancellationToken at session creation, wire it through AgentRuntime::set_interrupt_token exactly once per turn, (3) add tokio::select! on CancellationToken to both SSE parse loops (openai.rs, anthropic.rs), (4) build PermissionHook→ACP bridge via session/request_permission with well-defined notification schema and Always Allow persistence across turns, (5) add 30s timeout + cancel preemption to all agent→client RPCs including the cancel-during-permission edge case, (6) document cooperative-cancel design in a module-level doc comment. If scope proves too large, split into P4a (cancel endpoint + stream abort + token wiring + transcript repair + doc: C0, C1, C2, C3, C6, C7) and P4b (permission bridge + RPC timeout + edge cases: C4, C5).

## Criteria
- [C0] session/cancel is a registered ACP JSON-RPC method that accepts params: {sessionId: string} and returns {cancelled: true} on success. It returns a 404 error code for an unknown session ID and a 400 error code for missing or invalid parameters (e.g. wrong type for sessionId).
  - how: Unit test: (1) call session/cancel with a valid session ID from a freshly created session → assert response {cancelled: true}, (2) call session/cancel with a non-existent session ID → assert error response with code 404, (3) call session/cancel with params: {} (missing sessionId) → assert error response with code 400, (4) call session/cancel with params: {sessionId: 123} (wrong type) → assert error response with code 400.
- [C1] Session creation (session/new) materializes a tokio_util::sync::CancellationToken and stores it on the session. session/cancel fires that token. The same token instance is passed to AgentRuntime::set_interrupt_token exactly once at the start of each turn, and the same token is observable in both provider stream functions (openai.rs, anthropic.rs) and ACP client RPC calls via the AgentRuntime context. After a turn ends (regardless of finish reason), a fresh CancellationToken is created for the next turn — a cancel only affects the currently in-flight turn, never a future one.
  - how: Unit test: (1) Create a session via session/new, capture the CancellationToken from the session state via a test-only accessor, (2) start a turn with session/prompt, assert set_interrupt_token was called with a token that, when cancelled, causes the provider stream's select! branch to fire — confirming they share the same cancellation tree, (3) call session/cancel, assert the captured token is in cancelled state, (4) after the turn completes (AgentEvent::TurnFinished), start a second turn, capture the token again, assert it is a fresh (non-cancelled) token distinct from the first turn's token. Grep src/acp/server.rs for set_interrupt_token and assert exactly one call site per turn.
- [C2] session/cancel immediately aborts the in-flight LLM stream at the transport level (reqwest::Response is dropped, TCP connection closes), and the turn completes with stopReason: 'cancelled' without waiting for remaining SSE chunks.
  - how: E2E test: (1) Start session/new, (2) send session/prompt with a prompt that triggers a long LLM response, (3) after receiving the first agent_message_chunk, immediately send session/cancel, (4) measure wall-clock time from cancel-send to session/update with stopReason='cancelled' — must be <1s (not limited by the provider's chunk interval), (5) verify no additional agent_message_chunk notifications arrive after the cancel. Unit test: mock an SSE stream with a 5s delay between chunks, cancel after first chunk, assert stream returns Err(Cancelled) within 100ms.
- [C3] After a cancelled turn, the transcript remains structurally valid: every assistant message containing tool_calls has a corresponding tool result message (failed in-flight tool calls get a synthetic error tool_result with content indicating cancellation), so session/load replays without orphaned tool_calls and subsequent turns work correctly.
  - how: E2E test: (1) Create session, (2) send session/prompt with a tool-calling prompt (e.g. 'read src/main.rs then edit it'), (3) cancel after the first tool_call notification but before tool_call_update, (4) call session/load and capture all replayed notifications, (5) walk the replayed message sequence and assert: for every notification with a tool_call, there exists a subsequent tool_call_update with status='failed' and a synthetic error result containing 'cancelled' or 'interrupted' in its text. Unit test: load the transcript.jsonl from the cancelled session, iterate Role::Assistant messages that have tool_calls, verify each tool_call.id has a matching Role::Tool message immediately following.
- [C4] When the agent invokes a tool that requires permission (Edit, Bash, Write), the ACP client receives a session/request_permission notification matching the JSON-RPC shape {jsonrpc:"2.0", method:"session/request_permission", params:{sessionId, requestId, toolName, arguments}} — so the client can parse it deterministically. The agent blocks until the client responds with Allow, Deny, or Always Allow. On Allow, the tool executes normally. On Deny, the tool is rejected with a permission-denied error and the agent sees it as a tool failure. On Always Allow, subsequent invocations of the same tool kind skip the permission prompt for the remainder of the session (the exemption list is an in-memory hash set on the session, keyed by toolName, that survives turn boundaries).
  - how: E2E test (ACP client script): (1) Create session with a PermissionHook that requires confirmation for Edit, (2) send session/prompt: 'edit Cargo.toml', (3) assert session/request_permission notification arrives matching the schema {jsonrpc:"2.0", method:"session/request_permission", params:{sessionId, requestId, toolName, arguments}} — verify toolName='Edit' and arguments contain the file path, (4) capture requestId, send response with decision='Deny' → assert tool_call_update with status='failed' and error containing 'permission denied', (5) send another prompt: 'edit src/main.rs', (6) send response with decision='Allow' → assert tool_call_update with status='completed', (7) send response with decision='Always Allow' for the next edit → assert no further session/request_permission for Edit tools in this session, (8) start a new turn with another session/prompt that triggers an Edit tool → assert no session/request_permission notification fires, confirming the exemption list persists across turns. Unit test: verify the exemption hash set is stored on the session struct and survives TurnFinished/TurnStart boundaries.
- [C5] Every agent→client RPC (all methods registered under the client→agent RPC namespace at time of implementation, as discovered via Grep("rpc_methods") in the codebase) is protected by a CancellationToken that fires on session/cancel, plus a hard 30-second timeout. If session/cancel arrives while the agent is blocked on a session/request_permission awaiting client response, the CancellationToken fires and unblocks the RPC with a cancelled error (not a permission-denied error). If the client does not respond within 30 seconds, the RPC returns a timeout error and the agent does not hang indefinitely.
  - how: Unit tests: (1) Spawn an ACP server, (2) issue an RPC request (use any method from the namespace discovered via Grep("rpc_methods")), (3) sleep the mock client without responding, (4) assert the calling future resolves with a timeout error in 30s ± 2s (use tokio::time::timeout wrapping the assertion to avoid test hang), (5) repeat with session/cancel sent at t=5s and assert the RPC resolves with a cancelled error at t≈5s (not 30s) — verify the CancellationToken is the same one wired to session/cancel, so cancel preempts the 30s deadline, (6) cancel-during-permission edge case: (a) create session with PermissionHook requiring confirmation, (b) send prompt that triggers a tool needing permission, (c) assert session/request_permission notification fires, (d) before sending Allow/Deny, call session/cancel, (e) assert the permission RPC resolves with Error::Cancelled (not Error::PermissionDenied) within 200ms of cancel, (f) assert the turn ends with stopReason='cancelled'.
- [C6] src/acp/server.rs contains a module-level doc comment (//!) section titled 'Cooperative Cancel' that explains: (a) the two-phase abort design — Phase 1 cancels the LLM stream at the SSE read loop via tokio::select! on CancellationToken, Phase 2 cancels in-flight agent→client RPCs via the same CancellationToken, (b) how the CancellationToken flows from session/cancel → AgentRuntime::set_interrupt_token → provider stream functions and RPC calls, (c) the guarantee that after cancel, the transcript is repaired (tool_call↔tool_result pairing) before the turn ends.
  - how: Open src/acp/server.rs. Verify: (1) a module-level doc comment (//! at the very top of the file, before any use statements, fns, or structs) with a heading matching /Cooperative Cancel/i, (2) the text mentions 'tokio::select!' and 'CancellationToken', (3) the text describes both the SSE read loop level and the ACP client RPC level, (4) the text references transcript repair / tool_call-tool_result pairing guarantee, (5) the text explains the single-token-per-turn lifecycle (fresh token each turn). No automated test needed — this is a documentation gate verified by code review.
- [C7] Both src/llm/openai.rs::ChatProvider::stream and src/llm/anthropic.rs::ChatProvider::stream use tokio::select! on the CancellationToken (passed via AgentRuntime context) alongside the SSE chunk read. When the token fires, the reqwest::Response is dropped (closing the TCP connection), the stream returns Err(Error::Cancelled), and run_inner translates Error::Cancelled → FinishReason::Cancelled.
  - how: Unit test per provider: (1) Create a CancellationToken, (2) start a ChatProvider::stream call with a real or mocked SSE endpoint that sends chunks with a 2s interval, (3) after receiving the first chunk, fire the CancellationToken from another task, (4) assert the stream's next poll returns Poll::Ready(Some(Err(...))) where the error is Error::Cancelled within 200ms of token fire (not waiting for the 2s chunk interval). Grep both files for 'tokio::select!' to confirm the pattern exists. Integration test: create a mock ChatProvider whose stream method yields exactly one chunk then returns Err(Error::Cancelled), run it through run_inner, and assert the final AgentEvent is AgentEvent::TurnFinished { stop_reason: FinishReason::Cancelled }.
