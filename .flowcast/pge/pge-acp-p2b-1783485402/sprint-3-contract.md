# Contract: Sprint 3 — tool_call notifications + ToolKind + live status tracking (rev 2)

Wire the AcpBridge to emit tool_call (pending → in_progress) and tool_call_update (completed/failed) notifications. Add a default kind() → ToolKind method on the Tool trait with per-tool overrides. Extract locations metadata from tool arguments. Define ToolKind enum in src/acp/ with serde snake_case serialization.

## Criteria
- [S3-C1] AcpBridge emits a `tool_call` SessionUpdate notification (sessionUpdate: "tool_call") with status `pending` when an AgentEvent::ToolCall arrives, containing toolCallId, title (tool name), and the ToolKind derived from the tool's kind() method. The notification is emitted immediately (not buffered like text chunks).
  - how: Unit test: create AcpBridge, emit AgentEvent::ToolCall { name: "Read", id: "tc-1", arguments: "{\"path\":\"/tmp/f.txt\"}", step: 0 }, drain receiver. Assert one notification with method=session/update, params.update.sessionUpdate="tool_call", params.update.toolCallId="tc-1", params.update.title="Read", params.update.kind="read", params.update.status="pending". Verify the notification arrives before any TurnFinished event.
- [S3-C2] AcpBridge immediately follows the `pending` tool_call with an `in_progress` tool_call_update (synthesized internally by the bridge — there is no corresponding AgentEvent), then on AgentEvent::ToolResult emits `completed` (or `failed` if is_error=true) with the tool output content, so the full lifecycle is pending → in_progress → completed/failed.
  - how: The bridge synthesizes in_progress internally (no corresponding AgentEvent) by emitting a tool_call_update immediately after the pending tool_call. Unit test: emit ToolCall(id=tc-1), then ToolResult { id: "tc-1", name: "Read", output: "file contents", is_error: false, step: 0 } → TurnFinished. Drain receiver. Assert three tool notifications in order: (1) tool_call status=pending, (2) tool_call_update status=in_progress, (3) tool_call_update status=completed with content.type="content" and content.content.text="file contents". Repeat with is_error=true and assert status=failed. All three share the same toolCallId.
- [S3-C3] A new `ToolKind` enum is defined in `src/acp/` (not in `src/tools/registry.rs`, to avoid coupling tool registry to ACP types). Variants: `Read | Edit | Write | Execute | Search | Fetch | WebSearch | Other`, with `#[derive(Serialize)]` using `#[serde(rename_all = "snake_case")]`. The `Tool` trait in `src/tools/registry.rs` gains a default `fn kind(&self) -> ToolKind` method returning `ToolKind::Other`. Each concrete tool overrides: Read→Read, Edit→Edit, Write→Write (independent kind, not reused from Edit — Write is full overwrite, Edit is incremental patch, semantically distinct and rendered differently by ACP clients), Bash→Execute, Glob→Search, WebFetch→Fetch, WebSearch→WebSearch (independent kind, not merged with Glob's Search — spec requires "web_search" to distinguish file search from web search UI cards). All other tools (MCP, sub_agent, facts, etc.) inherit the default `Other`.
  - how: Grep `impl Tool for` across src/tools/ and verify each known tool either overrides kind() or inherits Other. Unit test: create a stub tool with no kind() override, assert kind() == ToolKind::Other. Integration test: iterate ToolRegistry::build_standard_tools(), assert each tool's kind() matches expected mapping (Read→Read, Edit→Edit, Write→Write, Bash→Execute, Glob→Search, WebFetch→Fetch, WebSearch→WebSearch, all others→Other). Verify ToolKind serializes as snake_case: ToolKind::WebSearch → "web_search". Run `cargo test --lib` — all pass.
- [S3-C4] The AcpBridge receives tool→kind() mapping via a registry lookup (or a pre-built HashMap) so it can include the correct `kind` field in tool_call notifications without hardcoding tool-name→kind mappings.
  - how: The bridge's `emit` method on ToolCall looks up the tool name in an internal HashMap<String, ToolKind> (populated at bridge construction time from the ToolRegistry). Unit test: bridge with {"Read": ToolKind::Read} → emit ToolCall(name="Read") → notification has kind="read". Bridge with empty map → emit ToolCall(name="UnknownTool") → notification has kind="other".
- [S3-C5] A `locations` extraction function (`extract_locations(tool_name: &str, arguments: &Value) -> Vec<ToolCallLocation>`) reads tool arguments and returns ToolCallLocation entries. Read extracts `path`; Edit extracts `file_path` with optional `start_line`→line; Write extracts `file_path`; Glob extracts `pattern`; WebFetch extracts `url`; Bash extracts `cwd` as a path location when present, returns empty Vec when absent.
  - how: Unit test: Read({"path":"/tmp/f.txt"}) → [{"path":"/tmp/f.txt"}]. Edit({"file_path":"/tmp/f.txt","old_string":"x","new_string":"y"}) → [{"path":"/tmp/f.txt"}]. Edit with start_line → includes line field. Write({"file_path":"/tmp/new.txt"}) → [{"path":"/tmp/new.txt"}]. Glob({"pattern":"/src/**/*.rs"}) → [{"path":"/src/**/*.rs"}]. WebFetch({"url":"https://ex.com"}) → [{"path":"https://ex.com"}]. Bash({"command":"ls","cwd":"/tmp/work"}) → [{"path":"/tmp/work"}]. Bash({"command":"ls"}) → []. Malformed args (missing fields) → empty Vec with no panic.
- [S3-C6] The initial `tool_call` (pending) notification includes the extracted `locations` array. Subsequent in_progress and completed tool_call_update notifications do NOT repeat locations.
  - how: Unit test: emit ToolCall(name="Read", args with path) → assert first notification (pending) has params.update.locations array with the path. Then assert the in_progress and completed notifications have no locations field (or it is absent/null). Locations are sent exactly once.
- [S3-C7] Multiple concurrent tool calls in a single turn each get independent toolCallIds, status tracking, and correct notification ordering. The bridge maintains per-call state so two parallel tool calls don't interfere and results can arrive out-of-order.
  - how: Unit test: emit ToolCall(id=tc-1)+ToolCall(id=tc-2) back-to-back, then ToolResult(id=tc-2, is_error=false)+ToolResult(id=tc-1, is_error=true) → TurnFinished. Assert 8 tool notifications in sequence: tc-1 pending, tc-1 in_progress, tc-2 pending, tc-2 in_progress, tc-2 completed, tc-1 failed. Each notification has the correct toolCallId and status.
- [S3-C8] The bridge's `_ => {}` catch-all in `emit()` is replaced with explicit match arms for ToolCall and ToolResult. Non-tool events (Latency, Usage, PlanMode*) remain silently discarded.
  - how: Code review: `match event` in AcpBridge::emit() has dedicated `AgentEvent::ToolCall { .. }` and `AgentEvent::ToolResult { .. }` arms. Unit test: emit Latency event → no notification produced. Emit Usage event → no notification produced.
- [S3-C9] All existing Sprint-2 tests continue to pass — agent_message_chunk streaming, stopReason, messageId stability, multi-turn context — are not broken by the addition of tool_call notifications.
  - how: Run `cargo test --lib acp` — all tests pass. Any test that asserts exact notification count (e.g. `lines.len()` == N) must be updated if the test path now produces additional tool_call notifications. The `session_prompt_emits_agent_message_chunk_and_stop_reason` test must still find agent_message_chunk with text content and stopReason=end_turn among the notifications.
- [S3-C10] `cargo test --workspace`, `cargo clippy --all-targets --all-features -- -D warnings`, and `cargo fmt --all -- --check` all pass clean with zero warnings and no new `unwrap()`/`expect()` in product code.
  - how: Run the full quality gate suite from workspace root. All three commands exit 0. No clippy warnings. No `unwrap()` or `expect()` introduced in non-test code under src/acp/ or src/tools/registry.rs.
