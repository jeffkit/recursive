# Contract: Sprint 2 — session/new + session/prompt (text-only with streaming) (rev 2)

Core session lifecycle and text-only prompt streaming over ACP. session/new sandboxes to cwd. session/prompt streams agent_message_chunk notifications and finishes with session/update (stopReason: end_turn). All messages get stable content-hash messageIds. Internally, ContentBlock[] ↔ Message conversion and Event→session/update translation happen in a pure EventSink adapter, never touching run_inner.

## Criteria
- [S2-C1] session/new with valid cwd creates a session and returns { sessionId, ... }
  - how: Send initialize, then session/new { cwd: "/tmp/test-workspace" }. Assert response has a non-empty sessionId string and no error. Verify the session appears in the session registry (can be listed).
- [S2-C2] session/new enforces filesystem sandbox — agent cannot read or write outside cwd
  - how: Create session with cwd="/tmp/sandbox". Send prompt like 'read /etc/passwd'. Verify the agent either refuses (tool resolution fails) or the read returns an error. Verify no file outside /tmp/sandbox is accessed. Unit test: resolve_within is called on every fs tool path before the tool executes.
- [S2-C3] session/new rejects missing or non-existent cwd
  - how: Send session/new { cwd: "/tmp/nonexistent-deadbeef" }. Assert JSON-RPC error response with a meaningful error code and message (not a silent success or panic).
- [S2-C4] session/prompt with text content starts LLM processing and returns an acknowledgment
  - how: Send session/prompt { sessionId, prompt: [{ type: "text", text: "Hello" }] }. Assert the JSON-RPC result is received promptly (the request itself succeeds) and the agent begins processing. Verify by observing subsequent agent_message_chunk notifications.
- [S2-C5] Text streaming: agent_message_chunk notifications carry delta text chunks as the LLM generates
  - how: Send session/prompt with a prompt that produces nontrivial output (e.g. 'Explain recursion'). Accumulate all agent_message_chunk notifications received. Assert: (a) at least 3 chunks arrived, (b) concatenating all chunk deltas in order yields the full agent response, (c) no chunk carries tool_call content (that's Sprint 3).
- [S2-C6] agent_message_chunk notifications have a stable messageId that is consistent across chunks of the same message
  - how: Send session/prompt. Observe all agent_message_chunk notifications. Assert: all chunks belonging to the same agent turn share the same messageId. The messageId itself should be a hex string (content hash).
- [S2-C7] session/update notification arrives after streaming completes with stopReason="end_turn" and the completed message
  - how: Send session/prompt. Wait for streaming to finish. Assert: exactly one session/update notification arrives after the last agent_message_chunk, containing (a) stopReason: "end_turn", (b) the complete assembled message with the correct messageId matching the chunks, (c) the sessionId. No further agent_message_chunk arrives after session/update.
- [S2-C8] messageId is a stable content hash — identical content produces identical messageId across different sessions or calls
  - how: Create two sessions. Send the same prompt to both. Assert: the agent_message_chunk and session/update messages from both sessions have identical messageIds. Then send a different prompt; assert the messageId differs from the first. Unit test: content_hash(blocks) is deterministic and collision-resistant.
- [S2-C9] EventSink converts Recursive internal Event stream into ACP wire notifications (agent_message_chunk, session/update) without touching run_inner
  - how: Unit test: create a mock Event stream (text delta events → completion event). Feed it into the EventSink. Assert: (a) each text delta event produces one agent_message_chunk notification, (b) the completion event produces one session/update notification with stopReason, (c) the conversion is a pure mapping — no LLM call, no kernel dispatch. AST-level check: no import of run_core or RunCore in the acp/ module.
- [S2-C10] ContentBlock[] from ACP wire format converts to Recursive's internal Message type correctly
  - how: Unit test: construct a ContentBlock[] with text blocks. Pass through the adapter's conversion function. Assert: the output is a valid Message with role="user" and content matching the original blocks. Text blocks map to text content. Unknown block types produce an error (not a silent ignore).
- [S2-C11] sessionId uses existing session infrastructure from src/session/ — no parallel session management
  - how: Grep src/acp/ for any Session struct definition or session map. Assert none exists — all session state goes through SessionStore / AgentRuntime from src/session/. The session/new handler calls into SessionStore::create or equivalent existing API.
- [S2-C12] Session context (tool registry and system prompt) is initialized from session/new parameters and programmatically verifiable without LLM-dependent assertions
  - how: (a) Tool registry: send session/new with a specific tools parameter (e.g. only ["read_file", "write_file"]). Programmatically inspect the session's internal tool registry (not via agent prompt) — assert the set of registered tool names equals exactly what was passed in session/new. (b) System prompt: send session/new with system_prompt_override: "You are a Python expert.". Programmatically read the stored system prompt string from the session state — assert it equals "You are a Python expert." byte-for-byte. Neither check involves sending a prompt to the LLM or asserting on agent output.
- [S2-C13] session/prompt with an invalid or expired sessionId returns a JSON-RPC error
  - how: Send session/prompt { sessionId: "nonexistent-session-id", prompt: [{ type: "text", text: "hello" }] }. Assert the response is a JSON-RPC error (contains error.code != 0 and error.message) rather than a success result. The error message should indicate the session is unknown or expired.
- [S2-C14] session/prompt with an empty or missing prompt array returns a JSON-RPC error
  - how: Test both cases: (a) session/prompt with prompt: [] (empty array), (b) session/prompt with the prompt field completely omitted from params. Assert both return JSON-RPC errors with meaningful messages (e.g. 'prompt must be a non-empty array'). No LLM processing or notifications should be triggered.
- [S2-C15] session/prompt on a terminated (closed/deleted) session returns a JSON-RPC error
  - how: Create a session, terminate/close/delete it, then send session/prompt on the same sessionId. Assert JSON-RPC error response indicating the session is no longer active (e.g. 'session terminated' or equivalent). No notifications should be emitted.
- [S2-C16] Session termination (close/delete/dispose) releases resources and removes the session from the registry
  - how: Create a session, send a prompt, wait for completion. Then send a close/delete/dispose request for that sessionId. Assert: (a) the close request returns success, (b) the session is no longer present in the session registry (listing sessions does not include it), (c) subsequent session/prompt on that sessionId returns an error (per S2-C15). If a TTL/expiry mechanism is implemented, also assert that idle sessions past their TTL are automatically cleaned up without explicit close.
- [S2-C17] Notifications (agent_message_chunk and session/update) conform to JSON-RPC 2.0 notification wire format
  - how: Capture the raw JSON of every notification emitted during a session/prompt exchange. For each notification, assert: (a) it is a JSON object with exactly the keys "jsonrpc", "method", and "params" — must NOT contain an "id" key, (b) jsonrpc equals the string "2.0", (c) method is one of "agent_message_chunk" or "session/update", (d) params is an object containing the notification payload (not null, not an array). Validate this for both notification types across at least one full session/prompt cycle.
